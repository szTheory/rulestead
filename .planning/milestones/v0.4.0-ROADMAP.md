# Roadmap: v0.4.0 - Experimentation & Analytics

## Phases

- [x] **Phase 16: Experimentation Core** - Extend the evaluation engine and data model to support formal A/B test experiments on top of existing flags.
- [x] **Phase 17: Analytics & Metrics Ingestion** - Build the ingress and storage mechanisms for impressions and conversion events.
- [x] **Phase 18: Experimentation UI & Reporting** - Expose experimentation lifecycles and calculated metrics to operators in the Admin UI.

## Phase Details

### Phase 16: Experimentation Core
**Goal**: The runtime engine can deterministically assign contexts to experiment variations, honoring sticky assignments and active experimentation periods.
**Depends on**: None
**Requirements**: EXP-01, EXP-02, EXP-03
**Success Criteria** (what must be TRUE):
  1. Operators can define an experiment with a target audience, percentage rollouts, and a control/holdout group.
  2. The runtime uses a distinct bucketing salt/seed for experiments so that a context's assignment is stable but orthogonal to normal flag bucketing.
  3. Experiments have explicit Active and Inactive states that govern whether traffic is routed to variations.
**Plans**: None yet

### Phase 17: Analytics & Metrics Ingestion
**Goal**: The system can ingest and store the telemetry necessary to calculate experimentation results.
**Depends on**: Phase 16
**Requirements**: ANA-01, ANA-02
**Success Criteria** (what must be TRUE):
  1. The platform reliably captures and stores evaluation "impressions" when an experiment variation is served.
  2. The `rulestead` package exposes a new API (e.g., `Rulestead.track(context, "event_name", metadata)`) for the host app to report business events.
  3. Events and impressions can be joined accurately based on the evaluation context.
**Plans:** None yet

### Phase 18: Experimentation UI & Reporting
**Goal**: Product managers and operators can view the results of A/B tests and monitor for regressions in the Admin UI.
**Depends on**: Phase 16, Phase 17
**Requirements**: ANA-03
**Success Criteria** (what must be TRUE):
  1. The Admin UI has a dedicated section for viewing active and historical experiments.
  2. Operators can see calculated conversion lifts and statistical significance for active experiments.
  3. The UI warns operators of unexpected error rates (guardrail metrics) associated with specific variations.
**Plans:** None yet
