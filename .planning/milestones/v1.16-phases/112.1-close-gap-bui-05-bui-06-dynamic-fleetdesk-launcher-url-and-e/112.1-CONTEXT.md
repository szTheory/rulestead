# Phase 112.1: Close gap: BUI-05/BUI-06 - dynamic FleetDesk launcher URL and evidence - Context

**Gathered:** 2026-06-12 (assumptions mode)
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 112.1 is an inserted v1.16 gap-closure phase for BUI-05/BUI-06. It fixes the concrete audit blocker where Phoenix-owned demo launcher links still point at `http://localhost:3000` even though compose can allocate and export a different FleetDesk frontend URL. It also adds targeted browser evidence that the launcher reaches the live selected FleetDesk origin.

This phase may touch demo backend URL plumbing, Phoenix launcher/layout links, targeted tests/evidence, and phase verification artifacts needed for this gap. It must not reopen brand direction, palette/logo decisions, runtime APIs, schemas, release workflow, component framework, or `rulestead_admin` publishing posture.
</domain>

<decisions>
## Implementation Decisions

### Dynamic URL Source

- **D-01:** Treat `DEMO_FRONTEND_URL` as the canonical external FleetDesk URL for the Phoenix demo backend, with `http://localhost:3000` only as the unaided local fallback.
- **D-02:** Read the URL through Phoenix runtime/application config, not by deriving from `DEMO_FRONTEND_PORT`, Phoenix endpoint config, request origin, CORS settings, or a release config-provider layer.
- **D-03:** Keep the selected URL single-source across compose, backend rendering, and Playwright evidence so direct-port and proxy host modes cannot drift.

### Phoenix-Owned Surfaces

- **D-04:** Fix both Phoenix-owned FleetDesk entry points: the launcher card in `home.html.heex` and the shared layout navigation link in `layouts.ex`.
- **D-05:** Prefer explicit Phoenix assigns/attrs for the URL over a global plug, implicit template helper, or new UI component abstraction. This keeps the two-link demo gap visible and testable without adding magic state.
- **D-06:** Update launcher copy that currently says "port 3000" so the UI does not contradict dynamic URL behavior.

### Browser Evidence

- **D-07:** Extend the existing compose-backed Playwright brand evidence instead of creating a new broad visual-regression program.
- **D-08:** Assert both rendered FleetDesk links have an `href` matching the normalized live `DEMO_FRONTEND_URL`.
- **D-09:** Add at least one click-through assertion from the Phoenix launcher to FleetDesk so the user path, not only independent page rendering, is proven.
- **D-10:** Add at least one build/release admin URL, ideally `/admin/flags/fleet-map-v2/rollouts?env=staging`, to the curated brand evidence matrix if Phase 112.1 plans include the audit warning alongside the dynamic-link blocker.

### Scope Boundary

- **D-11:** Keep Phase 112.1 to narrow gap closure: URL plumbing, link rendering, targeted browser/backend tests, and verification/planning truth.
- **D-12:** Explicitly exclude FleetDesk rebranding, Rulestead palette/logo/token changes, admin UI polish, public API/schema/config contract changes, release workflow edits, component framework adoption, `rulestead_admin` publish prep, and unrelated formatting cleanup.

### the agent's Discretion

The planner may choose the exact helper/module name for backend URL access, provided the result is explicit, small, testable, and does not hide host-app URL state in unrelated global plumbing.
</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets

- `scripts/demo/compose-env.sh` already computes and exports `DEMO_FRONTEND_URL` from the selected frontend origin.
- `scripts/demo/verify.sh` already passes `DEMO_BACKEND_URL` and `DEMO_FRONTEND_URL` into the frontend Playwright run.
- `examples/demo/frontend/playwright.config.ts` already stores the selected frontend and backend URLs in test metadata.
- `examples/demo/frontend/tests/brand-ui-evidence.spec.ts` already covers the Phoenix launcher and FleetDesk host-brand surfaces, making it the natural place for the targeted dynamic-link proof.

### Established Patterns

- v1.16 keeps Rulestead-owned demo chrome branded as Rulestead while FleetDesk remains a distinct host/example product.
- Evidence should be curated and machine-asserted, with screenshots as supporting artifacts, not brittle pixel baselines for every route.
- Phoenix rendering should use explicit assigns/components for view state rather than hidden global state when the data is needed by a small number of surfaces.

### Integration Points

- Backend runtime config must receive the selected FleetDesk URL from compose/dev env.
- `PageController` and/or template rendering must expose `@fleetdesk_url` to `home.html.heex`.
- `Layouts.app` must accept an explicit FleetDesk URL attr or otherwise receive the same configured URL without hardcoding.
- Playwright must compare rendered link `href`s against the live `DEMO_FRONTEND_URL`, accounting for normal URL normalization.
</code_context>

<specifics>
## Specific Ideas

- Prefer the app/runtime config key name `:fleetdesk_frontend_url` or a similarly explicit demo-local key.
- Keep links same-tab and accessible; this is a normal demo navigation path, not a marketing outbound link.
- If a backend test is added, use an overridden non-3000 URL to prove the fallback is not masking the bug.
- Playwright failure output should make expected vs actual origin clear enough for CI triage.
</specifics>
