# Plan 62-03: Protected-Environment Routing And Store Integration

<objective>
Complete governed execution routing: protected environments auto-submit change requests; non-protected environments direct advance — both through orchestrator with Fake/Ecto parity per CONTEXT D-04, ROL-06.
</objective>

<threat_model>
| Threat | Severity | Mitigation |
|--------|----------|------------|
| T-62-07: policy_bypass skips CR in production | critical | approval_requirement check before direct advance |
| T-62-08: Auto-approve CR in protected env | critical | submit_change_request only; no approve_change_request call |
| T-62-09: CR snapshot missing window context | medium | Include signal_facts + monitoring_window bounds in snapshot |
| T-62-10: Fake/Ecto protected routing drift | high | Shared orchestrator branch; contract tests on both |
</threat_model>

<task id="62-03-01" type="execute">
  <name>Protected-environment change request branch in orchestrator</name>
  <read_first>
    - rulestead/lib/rulestead/admin/authorizer.ex
    - rulestead/lib/rulestead/store/ecto.ex (submit_change_request, execute_bounded_governed_action advance ~4627)
    - .planning/phases/62-orchestration-and-governed-execution/62-CONTEXT.md (D-04)
  </read_first>
  <action>
In `RolloutAutoAdvance.execute_scheduled_tick/3`, after eligibility `:eligible`:

1. Build resource map: `%{"type" => "flag", "key" => flag_key}`
2. Call `Authorizer.approval_requirement(system_scheduler_actor, :advance_rollout, resource, environment_key)`
3. When `approval_requirement["change_request_required?"] == true`:
   - Build `Command.SubmitChangeRequest` with:
     - `governed_action: :advance_rollout`
     - `environment_key`, `resource_type: "flag"`, `resource_key: flag_key`
     - `command_snapshot`:
```elixir
%{
  "rollout" => %{
    "rule_key" => rule_key,
    "stage" => policy.next_stage,
    "percentage" => policy.next_percentage,
    "monitoring_window_started_at" => evaluated_at_iso,
    "monitoring_window_ends_at" => ends_at_iso
  },
  "signal_facts" => resolved_facts
}
```
     - `approval_requirement: approval_requirement` (full snapshot)
     - `metadata: %{"source" => "guardrail_automation", "scheduled_execution_id" => id, "observation_window_ends_at" => ..., "eligibility" => ...}`
     - `actor: system_scheduler_actor`
     - `reason: "auto_advance eligible; change request required in protected environment"`
   - `store.submit_change_request/1`
   - Return `{:ok, %{outcome: :change_request_submitted, change_request: cr}}`
4. When `change_request_required?` is false:
   - Return `{:ok, advance_command}` struct for direct `advance_rollout/1`

System scheduler actor:
`%{"id" => "system:scheduler", "type" => "system", "display" => "Scheduler"}`

Never call `approve_change_request` or `execute_change_request` from automation path.
  </action>
  <verify>
    <automated>cd rulestead && mix compile --warnings-as-errors</automated>
  </verify>
  <acceptance_criteria>
    - Orchestrator calls `Authorizer.approval_requirement/4` before direct advance
    - Protected path calls `submit_change_request/1` not `advance_rollout/1`
    - Return includes `outcome: :change_request_submitted` atom
    - No `approve_change_request` reference in rollout_auto_advance.ex
  </acceptance_criteria>
</task>
