Add helper `ensure_auto_advance_schema!/0` if needed (run latest migration).

**Tests (each iterates `@adapters`):**

1. **"upsert and fetch policy round-trip"**
   - Upsert enabled policy: `observation_window_seconds: 300`, `next_stage: "canary-50"`, `next_percentage: 50`
   - Fetch returns same fields
   - Unique key: flag `"aa-#{suffix}"`, env `"test"`, rule `"variant-split"`

2. **"evaluate blocked when policy disabled"**
   - Upsert `enabled: false`
   - Evaluate with healthy facts + closed window → `status: :blocked`, reasons include `"auto_advance_disabled"`

3. **"evaluate eligible when healthy after window close"**
   - Seed published rollout via `StoreFixtures` / `seed_published_rollout!` pattern from guarded_rollout_test
   - Upsert enabled complete policy
   - Evaluate with `monitoring_window_ends_at: ~U[2025-12-31 12:05:00Z]`, `evaluated_at: ~U[2025-12-31 12:06:00Z]`, healthy signal_facts
   - Assert `eligibility.status == :eligible`

4. **"evaluate blocked on pending_data before window close"**
   - Same policy; `evaluated_at` before `monitoring_window_ends_at` → `"monitoring_window_active"`

5. **"evaluate does not advance rollout stage"**
   - Record latest `GuardrailDecision` stage before evaluate (via fetch or advance)
   - Call evaluate when eligible
   - Assert stage unchanged / no new advance decision row with higher percentage (compare decision count or stage field)

6. **"ROL-07 guarded rollout rollback still works with auto-advance policy enabled"**
   - Enable auto-advance policy on same flag
   - Run abbreviated guarded rollout breach path from `guarded_rollout_test` (advance → evaluate healthy → advance → evaluate breach → rollback)
   - Assert rollback still occurs — auto-advance policy does not block hold/rollback

Use `reset_adapter!/1` and `adapter_suffix/1` helpers — copy from guarded_rollout_test if not in shared module.

Extend `store_fixtures.ex` only if a shared seed helper reduces duplication.
