# Requirements: v1.0.0 - General Availability (GA)

**Defined:** 2026-05-19
**Shipped:** 2026-05-21
**Core Value:** Phoenix teams can safely gate, roll out, and explain runtime decisions — booleans, variants, and remote config — with 15-minute quickstart, deterministic evaluation, and a calm admin UI that operators, support, and SRE can all trust at 3am.

## v1.0.0 Requirements

### API Stability & Documentation (API)

- [x] **API-01**: All internal modules are strictly marked with `@moduledoc false` to establish a clear public API boundary.
  Outcome: validated in Phase 26 verification.
- [x] **API-02**: Public API functions have comprehensive type specifications, and the project passes Dialyzer with zero warnings.
  Outcome: validated with documented tooling override. `rulestead` passes Dialyzer cleanly; `rulestead_admin` retains the known Dialyxir/Erlang baseline warning captured in `26-VERIFICATION.md`.
- [x] **DOC-01**: Hexdocs includes complete module documentation, architecture guides, and deployment recipes.
  Outcome: validated in Phase 26 verification.
- [x] **DOC-02**: A migration guide for FunWithFlags users is available in the documentation.
  Outcome: validated in Phase 26 verification.

### Comprehensive RBAC & Security (SEC)

- [x] **SEC-01**: The system defines explicit, static roles (Admin, Editor, Viewer) for accessing and mutating feature flags.
  Outcome: validated in Phase 27 verification.
- [x] **SEC-02**: Role-based access control is implemented using pure Elixir context-based boundaries without third-party authorization framework dependencies.
  Outcome: validated in Phase 27 verification.
- [x] **SEC-03**: The Admin UI and core API enforce RBAC policies, preventing unauthorized modifications to production flags.
  Outcome: validated in Phase 27 verification and Phase 28 browser proof.

### E2E Demo & GA Release (GA)

- [x] **GA-01**: A frictionless E2E demo environment is provided via Docker Compose, including Redis, DB, UI, and a sample client.
  Outcome: validated in Phase 28 verification.
- [x] **GA-02**: The demo environment includes a sample external frontend (e.g., Next.js) using the OpenFeature client to demonstrate cross-stack usage.
  Outcome: validated in Phase 28 verification.

## Future Requirements

### Deferred Beyond v1.0.0

- **EVAL-01**: New flag evaluation strategies beyond current boolean/variant semantics.
- **SEC-04**: Complex custom role definitions allowing arbitrary granular permissions.

## Out of Scope

| Feature | Reason |
|---------|--------|
| New Flag Evaluation Strategies | 1.0 is for locking down what exists. Adding new evaluation types resets the stability clock. |
| Complex Custom Role Definitions | Letting users define arbitrary roles with granular permissions is overkill for the first GA milestone. |
| Third-party RBAC dependencies | Using Ash or Permit for RBAC could cause version conflicts with host applications when Rulestead is mounted. |
