## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Lock icon gaps, inline reason spacing, badge padding trim |
| sm | 8px | Button-to-reason spacing, compact row actions, chip gaps |
| md | 16px | Default card padding, control spacing, disabled-state help text separation |
| lg | 24px | Section padding, action-group spacing, policy summary separation |
| xl | 32px | Summary-to-workspace gaps, split-pane detail spacing |
| 2xl | 48px | Major breaks between read context, capability state, and mutation sections |
| 3xl | 64px | Page-level breathing room inside the mounted host chrome |

Exceptions: 44px minimum hit area for icon-only controls and row actions; 1px hairlines for dividers, state rails, and production emphasis stripes.

Accessibility note:
- Disabled or restricted actions must still preserve hit-area sizing so the reason text remains discoverable on touch devices.
- Capability reason text sits directly below or adjacent to the blocked control; never behind hover-only disclosure.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#F3F6F8` | Page canvas, read surfaces, review body background |
| Secondary (30%) | `#E3EBF1` | Cards, capability summaries, disabled-action containers, env rails |
| Accent (10%) | `#2F6FED` | Primary CTA, active route/link state, current environment chip, focus ring |
| Destructive | `#C65A3A` | Admin-only destructive actions, fail-closed banners, protected-env warnings, destructive confirmations |

Accent reserved for: `Review permissions` CTA, active navigation, current environment locator, selected read-only drill-in row, focus-visible rings, and policy-explanation disclosures. Do not use accent for all links, all badges, or warning copy.

Additional semantic tones:
- Positive: `#2E7D5B` for allowed, healthy, or successfully reviewed states.
- Warning: `#C28A2E` for change-request-required, protected-environment, and elevated-review states.
- Neutral ink: `#22313D` for primary text; muted ink `#5E7282` for helper text, audit metadata, and policy explanations.
- Production emphasis: preserve the existing `data-env-tone` shell contract. Production headers use a thin destructive-tinted stripe plus plain-language authority copy.

Source/default rationale:
- Source: Phases 07 and 22 already locked the mineral neutral + controlled blue + restrained ember system.
- Source: Phase 27 keeps ember mapped to destructive/governance authority, not general accent, because the main job is making elevated permissions obvious without making the UI noisy.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Review permissions |
| Empty state heading | No restricted actions on this screen |
| Empty state body | Your current role can complete every action shown for this environment. Open another environment or a governed route to review elevated requirements. |
| Error state | We couldn't verify your access for this action. No change was applied. Refresh the page, confirm your environment scope, and retry after your host policy mapping is restored. |
| Destructive confirmation | Archive flag: "Archive this flag and preserve its audit history? Enter a reason before continuing."; Engage kill switch: "Type the flag key to engage the kill switch for this environment. This forces the default value until it is released."; Roll back rollout: "Roll back to the last approved exposure stage and record a reason."; Approve or execute governed change: "Confirm this reviewed change for {environment}. This action is audited under your operator identity."; Rotate webhook secret or change protected settings: "This updates control-plane configuration for {environment}. Enter a reason before saving." |

Additional wording rules:
- Teach only the canonical role vocabulary: `Viewer`, `Editor`, `Admin`.
- Legacy compatibility roles never appear in operator-facing UI copy.
- Authorization copy names the blocked action first, then the condition: `Requires Admin in production`, `Change request required in production`, `Editors can propose this change but cannot execute it`.
- Use `read`, `propose`, `approve`, `execute`, and `manage settings` consistently; do not collapse them into vague verbs like `access` or `modify`.
- Fail-closed copy must say that no change was applied.

## Interaction Contract

### Route-backed posture

- Preserve the existing mount-time `:access_admin` gate. Actors with no admin read scope at all are redirected out of the mounted surface.
- Read-oriented routes remain visible when they provide operational context: flag inventory/detail, rules read surfaces, simulation results, timelines, change-request detail, schedule detail, webhook visibility, diagnostics, and compare previews.
- Mutation-first routes with no useful read-only value may hard-deny or redirect: create flag, edit metadata, kill-switch mutation route, cleanup/archive flows, and other direct mutation entrypoints.
- URL-backed environment scope remains canonical. Authorization reasons must always refer to the currently selected environment, never to an implied global role.

### Capability rendering

- Capability display must be derived from the same closed action/resource/environment vocabulary as `Rulestead.Admin.Policy.can?/4` and authorizer enforcement.
- Viewers see live state, audit, diagnostics, and compare context, but mutation controls render as disabled with visible reason text.
- Editors see flag-scoped authoring controls in allowed environments and proposal-oriented alternatives in protected ones.
- Admins see elevated governance, protected-environment execution, control-plane settings, and recovery actions.
- Do not hide a blocked action if the operator benefits from understanding that it exists. Prefer disabled affordance + reason over omission for contextual mutation controls.

### Read vs mutate hierarchy

- Read context stays above the fold; capability or restriction messaging appears before the first mutation control.
- On governed screens, the order is: current state, policy summary, proposed change, allowed next steps, destructive or elevated actions last.
- Mutation controls are grouped by authority level:
  `Viewer unavailable` states first,
  `Editor proposal` states second,
  `Admin execute/manage` states last.
- A protected-environment warning is never the only sign of restriction; the relevant control itself must also carry disabled or gated state.

### Disabled and gated affordances

- Use native `disabled` controls where possible. If a semantic link or custom trigger must remain focusable, use `aria-disabled="true"` plus linked reason text via `aria-describedby`.
- Disabled styling uses reduced contrast plus a visible reason line; do not rely on opacity alone.
- Every blocked action reason is one sentence max and environment-specific.
- When a direct action is blocked but proposal is allowed, the primary control swaps from execute language to proposal language rather than showing two equally strong primaries.

### Governance and protected environments

- Production and other protected environments show plain-language policy state near the page header and again near elevated actions.
- Use warning tone for `change request required` states and destructive tone for `Admin only` or hard-denied states.
- Change-request flows must visually distinguish `can propose` from `can execute`.
- Approval, rejection, execution, schedule cancel/requeue, webhook secret rotation, and settings writes are always rendered as elevated actions.

### Server truth and audit posture

- Client-side capability projection is instructional only. Every action still submits to server enforcement, and denied attempts remain audit-visible.
- Post-submit denial messages use the error-state copy contract and do not imply partial success.
- Authorization failures never optimistically update the UI.

### Accessibility and mobile behavior

- Disabled reasons must be readable by screen readers without requiring hover.
- Focus order on desktop and mobile is: page summary, environment picker, policy summary, primary allowed action, secondary actions, then disabled actions with reasons.
- On narrow screens, action groups stack in authority order and reason text appears directly under each control.
- Production-state stripe and role chips are supplementary only; meaning must remain fully available in text.

## Component Inventory

Required shared components for Phase 27:

| Component | Purpose | Notes |
|----------|---------|-------|
| `rs-policy-state` | Environment-aware authority summary | Extend existing policy-state component with Viewer / Editor / Admin copy and protected-env reasons |
| `rs-banner` | Hard-deny, change-request-required, and fail-closed status banners | Reuse `OperatorComponents.banner/1` with warning and critical tones |
| `rs-status-list` | Capability matrix, route-level permissions, and elevated action summaries | Reuse existing status-list rows for `Allowed`, `Propose only`, `Admin only`, `Denied` states |
| `rs-summary-grid` | Role and environment fact summary | Reuse `rs-stat` cards for current role, environment, allowed scope, and governance state |
| `rs-card` / `section_card` | Read context, proposal context, and restricted-action grouping | Keep existing shell/card contract unchanged |
| `rs-badge` | Role chips and authority-state chips | Extend current badge tone system rather than introducing a new pill component |
| `rs-confirm-modal` | Elevated-action confirmation shell | Reuse confirm-modal pattern for typed-key, reason-required, and admin-only confirmations |

Extend existing components instead of replacing them:
- `Shell.page` remains the mounted page frame and environment emphasis source.
- `FlagComponents.section_card/1` remains the default wrapper for permission summaries and route-level action groups.
- `OperatorComponents.policy_state/1`, `banner/1`, and `status_list/1` remain the preferred base patterns for authorization teaching UI.
