# Phase 07 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | Phoenix.Component + Phoenix LiveView function components |
| Icon library | none locked yet; use inline SVG icons sized to the typography grid |
| Font | Sora headings, Inter body/labels, IBM Plex Mono for keys, versions, and trace data |

Notes:
- Source: `rulestead_admin` is a Phoenix LiveView package, not a React app, so shadcn is not applicable.
- Source: brand direction sets the type system as `Sora + Inter + IBM Plex Mono`.
- Default: keep the existing `rs-*` component namespace and extend it rather than introducing a second styling vocabulary.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline icon gaps, badge padding trim, diff line markers |
| sm | 8px | Form field gaps, status-chip spacing, compact metadata rows |
| md | 16px | Default field spacing, card internals, button padding |
| lg | 24px | Section padding, toolbar spacing, confirm modal body padding |
| xl | 32px | Screen-level grid gaps, two-column workspace spacing |
| 2xl | 48px | Major section breaks, summary-to-detail transitions |
| 3xl | 64px | Page top/bottom breathing room inside mounted host chrome |

Exceptions: 44px minimum hit area for icon-only controls and compact row actions; 1px hairlines for timeline dividers and diff boundaries.

Accessibility note:
- Any icon-only control must include an `aria-label` that names the action and expose a tooltip or visible text fallback on narrow/mobile layouts.

| Body | 16px | 400 | 1.5 |
| Label | 14px | 600 | 1.4 |
| Heading | 20px | 600 | 1.2 |

Additional contract:
- Use IBM Plex Mono at 14px / 400 / 1.4 for flag keys, snapshot versions, bucket values, and code/test-fixture export.
- Reserve Display for page titles and emergency-state banners only; do not use it inside dense audit rows.
- Keep all confirm modals at Body + Label only; the destructive state is signaled by color and wording, not oversized typography.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#F3F6F8` | App background, page canvas, modal body background |
| Secondary (30%) | `#E3EBF1` | Cards, side panels, env picker rail, audit row surfaces |
| Accent (10%) | `#2F6FED` | Primary CTA, active tab/step, current environment chip, focus ring, selected rollout ladder step |
| Destructive | `#C65A3A` | Kill switch engage/release, rollback, denied/destructive warnings only |

Accent reserved for: primary action buttons, current environment state, active navigation/tabs, selected rollout percentage chips, focus-visible rings, and draft/publish state that requires operator attention. Do not use accent for every link or every badge.

Additional semantic tones:
- Positive: `#2E7D5B` for safe/healthy states such as active, released, refresh-success.
- Warning: `#C28A2E` for potentially stale, risky rollout jumps, non-prod stripes, and redaction notices.
- Neutral ink: `#22313D` for primary text; muted ink `#5E7282` for helper text and metadata.
- Production emphasis: a thin destructive-tinted top border/stripe when `env=prod`; non-prod gets warning-tinted stripe. This extends the existing `data-env-tone` shell contract.

Source/default rationale:
- Source: brand posture calls for mineral neutrals, a controlled blue, and a restrained ember accent.
- Default: ember is mapped to destructive rather than general accent because Phase 7 is heavy on kill-switch and rollback risk states.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Run simulation |
| Empty state heading | No audit events yet |
| Empty state body | Changes will appear here after a publish, rollout step, kill switch action, or denied mutation. Start by running a simulation or publishing a reviewed draft. |
| Error state | We couldn't load the latest redacted operator data. Refresh the page or verify your environment access. If it persists, check snapshot health and policy configuration. |
| Destructive confirmation | Engage kill switch: "Type the flag key to engage the kill switch for this environment. This forces the default value until you release it."; Release kill switch: "Release the override and restore authored behavior for this environment?"; Roll back rollout: "Roll back to the last known good exposure stage and record a reason."; Publish risky rollout jump: "This jump increases exposure beyond the safe ladder. Review the preview and enter a reason before publishing." |

Additional wording rules:
- Use canonical domain terms from the field guide: `flag`, `rule`, `audience`, `rollout`, `kill switch`, `engage`, `release`, `audit event`, `targeting key`.
- Use `production`, `staging`, `development` in labels; use `prod`, `staging`, `dev` only in compact chips or URLs.
- Every mutation confirmation requires a reason field except non-prod restore/release actions, where reason is optional but still encouraged.
- Unauthorized states say what is blocked and why: `You can view this flag in Production, but your policy does not allow publishing here.`

## Interaction Contract

### Route-backed surfaces

- Add dedicated LiveView routes for `/admin/flags/:key/simulate`, `/admin/flags/:key/rollouts`, `/admin/flags/:key/kill`, `/admin/flags/:key/timeline`, and `/admin/audit`.
- Keep `?env=` as the canonical environment selector on every new Phase 7 route.
- The existing detail page remains the calm read surface and gains only summary/banner entry points into Phase 7 actions.

### Screen hierarchy

- Simulation screen: result summary first, trace details collapsed by default, fixture export visibly adjacent to the result.
- Rollouts screen: ordered-rule position context first, ladder suggestions second, publish controls last.
- Kill switch screen: current override state first, engage/release control second, recent events third.
- Per-flag timeline: reverse-chronological stream with redacted human-readable summaries first and raw diff disclosure second.
- Global audit: pinned high-impact events first, then filterable timeline.

### Mutation model

- Preserve the Phase 6 explicit draft/save/publish split for rollout editing.
- Small rollout edits may preview optimistically in-form, but publish, rollback, kill-switch engage, and kill-switch release are never optimistic.
- Every destructive or production-scoped mutation follows `preview -> confirm -> audit`.
- Typed-key confirmation is required for production kill-switch engage. Non-production uses a standard confirm modal with required reason.
- Risky rollout jumps are any publish action that skips the recommended ladder (`5 -> 25 -> 50 -> 100`) or increases exposure by more than one ladder step. They require an elevated warning panel in the confirm modal.

### Progressive disclosure

- Default visible summary fields for simulation/explain: matched rule or default path, value/variant, reason, bucket result, snapshot version, cache age.
- Hide per-condition pass/fail, bucket math inputs, and raw trace structure behind expandable sections.
- Show diff summaries in prose first; raw structured diff remains expandable.
- Keep advanced controls out of the detail hero. Link out to the dedicated route-backed screens.

### Environment and policy signaling

- Every page header keeps the environment picker on the right within the existing shell.
- Production pages show a destructive-toned stripe plus plain-language warnings on actions that mutate live state.
- Read-only policy states disable buttons and include inline explanatory text rather than disappearing controls entirely.
- Denied mutation attempts still surface in the timeline as redacted denied audit events.

### Feedback and async behavior

- Use inline status regions for draft-save and simulation completion.
- Long-running simulation preview loads should stream or chunk results and expose `Stop Sample Run` for sample-based runs.
- Never capture sockets inside async work; all async results must re-enter via message/assign patterns compatible with LiveView safety rules already locked for Phase 7.

### Mobile behavior

- On narrow screens, convert desktop two-column layouts to a single stack ordered as: summary, risk state, primary action, recent audit.
- Confirm modals remain full-width sheets on mobile with primary action pinned at the bottom.
- Timeline and diff rows collapse to summary + timestamp + expand action; actor and environment metadata move into the expanded state.

## Component Inventory

Required shared components for Phase 7:

| Component | Purpose | Notes |
|----------|---------|-------|
| `rs-banner` | High-signal state banners | Used for active kill switch, risky rollout jump, read-only policy warning |
| `rs-summary-grid` | 2-6 fact summary cards | Extends existing `rs-stat` pattern |
| `rs-trace-panel` | Simulation/explain summary + expandable detail | Monospace subrows for bucket/snapshot/test fixture |
| `rs-rollout-ladder` | Suggested `%` steps with selected/current state | Shows monotonic widening only |
| `rs-confirm-modal` | Reason-required confirm flow | Supports typed-key confirmation variant |
| `rs-audit-timeline` | Per-flag and global append-only event stream | Shared event row + filter bar + export affordance |
| `rs-diff-card` | Readable before/after diff | Summary first, raw structure optional |
| `rs-policy-state` | Disabled-action explanation | Inline on buttons, cards, and confirm dialogs |

Extend existing components instead of replacing them:
- `Shell.page` remains the shared page frame and env-tone source.
- `FlagComponents.section_card/1` remains the default surface wrapper for summary panels.
- `FlagComponents.stat/1` remains the base pattern for simulation and kill-switch fact tiles.
