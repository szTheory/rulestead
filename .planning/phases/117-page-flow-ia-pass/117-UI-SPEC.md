## Scope Guardrails

| Guardrail | Contract |
|-----------|----------|
| Navigation model | Preserve `RulesteadAdmin.Navigation`: Overview plus Build & release, Explain & diagnose, and Review & approve. Audiences, rollouts, audit, destructive actions, onboarding, denied, unavailable, and rare states are lenses inside those groups, not new top-level groups. |
| Route ownership | Phase 117 is a route-flow IA pass. Keep inventory filters/search, rules workspace state, kill-switch sequencing, home task composition, audience inventory, audit, explain, and simulate behavior route-owned unless evidence proves a stable reusable subpattern. |
| Foundation boundary | Reuse Phase 115 tokens, breakpoints, focus ring, reduced-motion, radius/elevation, and dense-content containment. Do not reopen foundations unless repeated route failures prove a shared regression. |
| Component boundary | Reuse Phase 116 primitives and composites. Do not treat audit/explain/simulate forms or route shells as hidden component-polish debt. |
| Evidence posture | Use deterministic matrix fixtures plus selected real mounted-admin route evidence. Screenshots are generated artifacts for human review, not checked-in baselines or pixel-diff gates. |
| Future-phase boundary | Phase 118 owns milestone-wide evidence closeout and durable guard extensions. Phase 117 should hand off a route evidence map and proof commands. |

## Spacing Scale

Declared values for route-owned IA adjustments (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px / `--rs-space-1` | Icon gaps, inline metadata gaps, compact focusable affordance spacing |
| sm | 8px / `--rs-space-2` | Compact row spacing, filter token gaps, inline control groups |
| md | 16px / `--rs-space-4` | Default route section spacing, form/control group spacing |
| lg | 24px / `--rs-space-6` | Route section padding, page-owned panel spacing, mobile stack gaps |
| xl | 32px / `--rs-space-8` | Major route group separation, first-answer to detail separation |
| 2xl | 48px | Large route-cluster breaks when page hierarchy needs clear separation |
| 3xl | 64px | Page-level separation only when preserving scan order on dense desktop routes |

Exceptions: Existing shipped component internals keep their current tokenized spacing. `--rs-touch-target-min` / 44px remains the exception for icon-only controls, shell controls, command-palette controls, destructive-flow controls, and compact mobile actions.

## Typography

Declare these four roles for route-owned headings, labels, body copy, evidence notes, and state copy. Existing component internals keep their shipped typography unless a route IA defect requires a local adjustment using these roles.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | `--rs-text-base` / 0.95rem | `--rs-weight-normal` / 400 | `--rs-leading-normal` / 1.5 |
| Label | `--rs-text-sm` / 0.86rem | `--rs-weight-semibold` / 600 | `--rs-leading-snug` / 1.35 |
| Heading | `--rs-text-lg` / 1.15rem | `--rs-weight-semibold` / 600 | `--rs-leading-tight` / 1.2 |
| Display | `--rs-text-2xl` / clamp(1.5rem, 2vw, 2rem) | `--rs-weight-semibold` / 600 | `--rs-leading-tight` / 1.2 |

Use only these two route-owned weights for new or revised IA surfaces: 400 and 600. Monospace values for flag keys, audience keys, fingerprints, raw audit details, JSON, and code samples use `--rs-font-mono` within Body or Label roles above.

## Color

Use the existing theme-scoped admin tokens. Phase 117 must preserve light, explicit dark, and system-dark behavior without introducing new palette values.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--rs-bg`, `--rs-surface` | Shell background, page body, main route panels, primary workflow surfaces |
| Secondary (30%) | `--rs-surface-muted`, `--rs-surface-faint`, `--rs-border`, `--rs-border-subtle`, `--rs-text-muted` | Cards, sidebar/nav, metadata rows, filters, secondary route context, empty/unavailable explanations |
| Accent (10%) | `--rs-primary` | Primary route action, active route/subnav indicator, selected command-palette option, visible focus emphasis, current workflow step |
| Destructive | `--rs-error`, `--rs-critical`, `--rs-error-soft`, `--rs-error-border` | Kill-switch, archive/delete, risky rollout, governed execute, blocked destructive, and danger confirmation states only |

Accent reserved for: the primary action in a route-owned workflow, active rail/subnav/current-step indicators, selected command palette option, focused shell/route controls, and existing route-active affordances. It is not for every link, every button, decorative highlights, or generic status coloring.

The existing Ember Copper `--rs-accent` remains a restrained brand and risky-state marker where components already use it: lockup route node, timeline/highlight details, risky-state emphasis, and authored-state markers. Do not broaden it into a second general interaction accent.

## Route Flow And IA Contract

| Area | Contract |
|------|----------|
| First-glance answer | Every reviewed route must expose the operator's first answer above or near the first viewport: what is happening, why it matters, and the next safe action. |
| Navigation grouping | Rail, home launcher, command palette, breadcrumbs, and contextual subnav must stay consistent with the grouped JTBD model. Do not add flat entity rails or new top-level Rulesets, Settings, Audit, Rollouts, or Destructive groups. |
| Route review artifact | Produce `117-FLOW-IA-REVIEW.md` or equivalent with columns for operator job, route cluster, page-owned surface, state coverage, finding, action, proof, and follow-on. |
| Inventory flow | Review search/filter hierarchy, tokenized search, suggestions, clear-query affordances, view tabs, result count, pagination relationship, stale cleanup prominence, and mobile card scan order. |
| Rules workspace | Review draft versus active status, publish/save/move/archive hierarchy, sidebar balance, missing-audience recovery, keyboard order through editor controls, and narrow viewport action reachability. |
| Kill switch | Review 3am decision order: current serving state, emergency evidence, reason, typed key, disabled/unavailable escalation, diagnostics link, audit history, and after-action return path. |
| Home overview | Review attention priority, grouped navigation consistency, onboarding/intermediate/advanced task balance, no-urgent-work copy, and command-palette alignment. |
| Audience inventory | Review inventory density, hidden/denied dependency placement, action visibility, archived/read-only distinction, and mobile table/list usability. |
| Audit/explain/simulate | Include in evidence. Edit only when route evidence shows a missing first answer, buried permalink/sample/context, unclear redaction, inaccessible raw detail, poor mobile/keyboard flow, or stranded audit row. |
| Long content | Long flag keys, audience keys, tenant/environment labels, audit reasons, raw details, and command labels must wrap or local-scroll inside their component/route boundary without page-level overflow. |

## Interaction Contract

| Interaction | Contract |
|-------------|----------|
| Keyboard path | Representative route evidence must cover rail/home navigation, command palette open/search/result selection, filters/forms, contextual subnav/tabs, and at least one destructive or review workflow. |
| Focus order | Focus must follow visual and decision order: route orientation, primary action or filter, main evidence, secondary detail, then destructive or advanced controls. Avoid focus jumps into low-priority sidebars before the first answer. |
| Command palette | Command palette grouping must derive from the same navigation source as rail/home. It must remain reachable by keyboard, expose grouped options, and keep selected option state visible. |
| Mobile/narrow | Mobile route order must prioritize first answer, primary safe action, critical evidence, then dense detail. Primary actions must remain reachable without horizontal page overflow. |
| Destructive flows | Keep preview -> confirm -> audit. Require reason where applicable, typed confirmation for high-risk actions, specific danger emphasis, explicit disabled/unavailable explanations, a non-danger back link, and audit handoff. |
| Forms and filters | Prefer semantic links, buttons, forms, `handle_params/3`, `push_patch`, streams, and route-owned form handling. Use DOM-aware JavaScript only for existing focus-heavy widgets such as the command palette. |
| Reduced motion | Route fixes must not add nonessential scale, translate, staged entrance, hover lift, or confirm-pop motion for reduced-motion users. State changes must remain visible without animation. |

## Evidence Contract

| Evidence target | Contract |
|-----------------|----------|
| Route set | Cover home/overview, flag inventory, rules workspace, kill switch, audience inventory, audit, explain, and simulate unless the plan records a narrower source-backed route set with equivalent FLOW coverage. |
| State coverage | Each reviewed route cluster must record happy-path, error, boundary, and rare-state coverage, using deterministic matrix fixtures where full route seed semantics would widen product scope. |
| ExUnit/Phoenix | Use source or LiveViewTest assertions for route-cluster mapping, fixture health, forbidden tooling boundaries, and source-owned contracts. |
| Playwright | Use browser evidence for focus order, keyboard flow, command palette behavior, mobile/narrow scan order, no horizontal overflow, key ARIA roles, and generated screenshot artifacts. |
| Themes | Route evidence should include light, dark, and system-dark where visual behavior differs. Do not create new theme tokens. |
| Viewports | Use desktop and mobile/narrow evidence. Preserve desktop density while proving mobile containment and reachable primary actions. |
| Screenshots | Write curated artifacts with filenames that include route cluster, theme, viewport, and special condition where applicable. Do not use `toHaveScreenshot`, `matchSnapshot`, pixelmatch, visual-diff tooling, or checked-in screenshot baselines. |
| Phase 118 handoff | Record proof commands, route coverage, intentional exceptions, and any evidence-triggered follow-on so Phase 118 can close VER requirements without rediscovering Phase 117 decisions. |

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Review Route Flow |
| Empty state heading | No route findings recorded |
| Empty state body | Add route evidence for this operator job before changing navigation or page hierarchy. |
| Error state | Route evidence could not be collected. Check the named route, fixture state, and demo backend URL, then rerun the focused route-flow proof. |
| Destructive confirmation | Kill switch / archive / delete / risky rollout: Confirm the affected flag or audience key, provide a reason, and keep `Return to route review` as the non-danger path. |

Additional copy rules:

- Use operator-specific verbs: `Review Route Flow`, `Open Diagnostics`, `Explain Decision`, `Simulate Actor`, `Inspect Audit Event`, `Return to Route Review`.
- Avoid generic one-word labels such as `Submit`, `Go`, or `OK` on new or revised route-owned controls.
- Empty, unavailable, denied, and blocked states must name what is missing, why the operator cannot proceed, and the next safe place to inspect.
- Destructive copy must identify the object, consequence, required evidence/reason, and return path.
- Audit/explain/simulate copy must remain support-safe and explicit about redaction, sample context, permalink state, and raw-detail availability.
