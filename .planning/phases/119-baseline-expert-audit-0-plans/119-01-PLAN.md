<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub API -> audit document | Live workflow, run, and branch-protection metadata is external mutable state entering a committed planning artifact. |
| Workflow/release source -> audit recommendations | CI YAML and release scripts define trust posture; misreading them could produce unsafe later recommendations. |
| Maintainer docs -> audit baseline | `MAINTAINING.md` may drift from live GitHub settings; the audit must distinguish documented intent from live enforcement. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-119-01 | Spoofing | `gh workflow list` / branch-protection API evidence | mitigate | Record the exact `gh` commands, repo `szTheory/rulestead`, collection date, and command result in `119-CI-CD-AUDIT.md`; do not treat undocumented live state as source truth without a `[VERIFIED: command]` tag. |
| T-119-02 | Tampering | Required-check and `release_gate` interpretation | mitigate | Compare `.github/workflows/ci.yml`, `scripts/ci/release_gate.sh`, and `MAINTAINING.md`; explicitly record `openfeature-companion` absent from `release_gate.needs` instead of changing workflow behavior in Phase 119. |
| T-119-03 | Repudiation | Branch protection mismatch | mitigate | Preserve both documented required checks and live API result text so later phases can explain whether a recommendation came from docs or live settings. |
| T-119-04 | Information disclosure | Release workflow secrets | mitigate | Name secret variables such as `HEX_API_KEY` only as identifiers; do not print secret values or request dashboard access. |
| T-119-SC | Tampering | npm/pip/cargo installs | accept | This plan performs no package-manager install tasks and adds no dependencies. |
</threat_model>
