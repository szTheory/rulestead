# Phase 114 - UI Design Contract

> Visual and interaction contract for the repo-native component matrix harness. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none - repo-native Phoenix LiveView harness using the existing Rulestead admin design system |
| Preset | not applicable |
| Component library | Phoenix.Component plus `RulesteadAdmin.Components.*` modules rendered from the demo host |
| Icon library | Existing admin component glyphs and inline affordances only; do not add an icon package |
| Font | `--rs-font-display` Sora for display/title roles, `--rs-font-sans` Inter for UI/body roles, `--rs-font-mono` IBM Plex Mono for keys, fingerprints, JSON, and code values |

Manual design-system source: `rulestead_admin/priv/static/css/rulestead_admin.css`, `RulesteadAdmin.Components.Shell.page/1`, and the reusable component modules under `rulestead_admin/lib/rulestead_admin/components/`.

No shadcn, Tailwind, React, Storybook, PhoenixStorybook, third-party registry, package install, public route, schema, release workflow, or `rulestead_admin` publish-prep change belongs in this phase.

## Scope Guardrails

| Guardrail | Contract |
|-----------|----------|
| Host placement | Matrix UI lives in the demo backend as a dev/test-only Phoenix LiveView route, for example `/dev/rulestead-admin/ui-matrix`. It must not be added to `RulesteadAdmin.Router.rulestead_admin/2`. |
| Source truth | Render real `RulesteadAdmin.Components.*` function components with deterministic assigns. Do not copy component HEEx into a static catalog. |
| Wrapper | Render the matrix inside `.rs-shell` through `RulesteadAdmin.Components.Shell.page/1` so theme, focus, command palette, rail, breadcrumbs, and admin CSS behavior are real. |
| Static fixtures | Keep `design-system.html`, `theme-control-harness.html`, and `theme-harness.html` available for token/theme guard assertions. They are supporting evidence, not the component contract. |
| Future-phase boundary | Foundation hardening is Phase 115, component polish/consolidation is Phase 116, page-flow IA is Phase 117, and milestone-wide evidence closeout is Phase 118. Phase 114 records defects it exposes unless a defect prevents the matrix from rendering. |

## Spacing Scale

Declared values for new matrix layout decisions (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline metadata gaps, small badge spacing |
| sm | 8px | Compact row spacing, inline control groups |
| md | 16px | Default matrix cell spacing, section stack spacing |
| lg | 24px | Matrix section padding, grouped example spacing |
| xl | 32px | Major group separation inside the shell body |
| 2xl | 48px | Large taxonomy bucket breaks |
| 3xl | 64px | Page-level break between overview and full matrix body when needed |

Exceptions: Existing admin components keep their current internal CSS tokens because Phase 114 renders the shipped component system without foundation hardening. The existing `--rs-touch-target-min` control-size exception remains valid for icon-only and compact shell controls because it preserves accessible hit targets.

## Typography

Declare these four roles for new matrix-specific chrome, labels, and fixture metadata. Existing rendered components retain their shipped internal typography.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | `--rs-text-base` / 0.95rem | `--rs-weight-normal` / 400 | `--rs-leading-normal` / 1.5 |
| Label | `--rs-text-sm` / 0.86rem | `--rs-weight-semibold` / 600 | `--rs-leading-snug` / 1.35 |
| Heading | `--rs-text-lg` / 1.15rem | `--rs-weight-semibold` / 600 | `--rs-leading-tight` / 1.2 |
| Display | `--rs-text-2xl` / clamp(1.5rem, 2vw, 2rem) | `--rs-weight-semibold` / 600 | `--rs-leading-tight` / 1.2 |

Do not introduce additional font families or new matrix-only font weights. Monospace values for flags, audience keys, fingerprints, raw audit details, and JSON/code samples use `--rs-font-mono` within the Body or Label roles above.

## Color

Use the existing theme-scoped admin tokens. The matrix must prove light, explicit dark, and system-dark behavior without introducing new palette values.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--rs-bg`, `--rs-surface` | Shell background, page body, matrix section surfaces |
| Secondary (30%) | `--rs-surface-muted`, `--rs-surface-faint`, `--rs-border`, `--rs-border-subtle`, `--rs-text-muted` | Matrix metadata rows, taxonomy labels, muted descriptions, card separators |
| Accent (10%) | `--rs-primary` | Primary matrix action, current section indicator, active nav/focus emphasis, selected command palette option |
| Destructive | `--rs-error`, `--rs-critical`, `--rs-error-soft`, `--rs-error-border` | Destructive fixture examples and danger submit examples only |

Accent reserved for: primary matrix action, active/current matrix section marker, focused shell/matrix control, selected command palette option, and existing route-active affordances already implemented by admin components. It is not for every link, every button, or decorative callouts.

The existing Ember Copper `--rs-accent` remains a restrained brand marker for the shipped lockup, timeline/highlight details, and risky-state emphasis where components already use it. Do not broaden it into a second general interaction accent in Phase 114.

## Visual Hierarchy and Interaction Contract

| Area | Contract |
|------|----------|
| Primary focal point | The first screen should make `Rulestead admin UI matrix` and the section index visible inside the real `.rs-shell`. The primary visual anchor inside each section is the real rendered admin component, not explanatory prose. |
| Secondary information | Source module, fixture state, operator lens, and evidence target metadata are secondary. Keep this metadata compact and visually quieter than the component example. |
| Matrix organization | Order sections by Phase 113 taxonomy: overview/shell, foundations reference rows, primitives, composites, page patterns/seeded flows, workflow states, then evidence/static-fixture links. |
| Section identifiers | Every major section must expose a stable `data-matrix-section` value so Playwright can assert representative coverage without relying on fragile copy. |
| Long labels and long keys | Long names, route labels, owner/team labels, flags, audience keys, fingerprints, JSON/code values, and audit reasons must wrap or local-scroll within their component boundary. They must not create page-level horizontal overflow. |
| Disabled/unavailable states | Disabled or unavailable controls must include nearby copy explaining why the action cannot proceed. Do not rely on opacity alone. |
| Destructive examples | Destructive examples show preview -> confirm -> audit expectations with a reason field, typed confirmation where applicable, danger emphasis, disabled/unavailable variants, and a specific return link. They must use fixture data only. |
| Keyboard and focus | Keep shell search, theme control, rail links, section anchors, task links, forms, command palette options, and destructive confirm controls keyboard reachable with visible focus. |
| Motion | Reduced-motion contexts must render the same content without relying on animation to communicate state. Do not add new nonessential staged or scale motion in Phase 114. |

## Matrix Content Contract

| Section | Required examples | Primary sources |
|---------|-------------------|-----------------|
| Overview and shell | Real shell chrome, long environment/tenant names, access policy state, command palette trigger, theme control, breadcrumbs, and rail links | `RulesteadAdmin.Components.Shell`, `RulesteadAdmin.Navigation` |
| Foundations reference rows | Token/theme/focus/static-fixture links and notes that static fixtures remain low-level guards | CSS token files and static fixtures |
| Primitives | Banners, page sections, record rows, detail grids, task links, signals, empty states, related links, summary grids, badges, pagination, stats, section cards, callouts, and flag subnav | `OperatorComponents`, `FlagComponents` |
| Composites | Rollout ladders, guardrail panels, rule editor cards, audit timeline/diff/raw detail, audience dependency/impact panels, governance/blast radius, simulate/explain traces | `RolloutComponents`, `RuleEditorComponents`, `AuditComponents`, `AudienceComponents`, `AudienceTraceComponents`, `GovernanceComponents`, `SimulateComponents` |
| Page patterns and seeded flows | Links or embedded route examples for flag inventory, rules, rollouts, audit/timeline, command palette shell context, and destructive preview/confirm/audit paths | Existing demo admin routes under `/admin/flags` |
| Workflow states | normal, dense, empty, loading, error, permission-denied, read-only, long-label, long-key, narrow-width/mobile, destructive-action, disabled, unavailable, focus, and keyboard-relevant cases | Phase 113 UI matrix contract |

Each section should make the fixture source obvious enough for later phases to file defects against the correct bucket, but should not become a documentation-heavy component catalog.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Review Matrix Evidence |
| Empty state heading | No matrix examples match this section |
| Empty state body | Add a deterministic fixture for the missing state before planning polish work for this bucket. |
| Error state | Matrix fixture failed to render. Inspect the named fixture helper, keep the static token fixtures available, and rerun the matrix smoke test after the assigns are fixed. |
| Destructive confirmation | Preview destructive fixture: Type the flag key before rendering the disabled or danger action example; use `Return to matrix overview` as the non-danger return link. |

Additional copy rules:

- Do not use generic one-word action labels for new matrix chrome.
- Use action-specific labels such as `Review Matrix Evidence`, `Open Rollout Example`, `Inspect Audit Detail`, `Return to Matrix Overview`, and `Render Destructive Preview`.
- Error and unavailable copy must name what is missing and where the maintainer should inspect next.
- Empty states must explain whether the absence is a valid fixture condition or a missing matrix example.

## Evidence Contract

| Evidence target | Contract |
|-----------------|----------|
| ExUnit/Phoenix | Add a route/component smoke test that proves the dev/test matrix route renders `.rs-shell`, the required section identifiers, and representative real component output. |
| Playwright themes | Visit the matrix in light, explicit dark, and system-dark contexts using the existing browser-context/localStorage pattern from `brand-ui-evidence.spec.ts`. |
| Playwright viewports | Visit the matrix in desktop and mobile/narrow viewports. Assert no horizontal page overflow with the existing `expectNoHorizontalOverflow(page)` shape. |
| Reduced motion | Include at least one reduced-motion browser context for the matrix and assert representative content remains reachable. |
| Keyboard/focus | Probe a small high-value path: shell search opens the command palette, command palette options are visible, and one matrix section/action can receive keyboard focus. |
| Screenshots | Write curated screenshot artifacts with filenames that include matrix section, theme, viewport, and reduced-motion where applicable. Do not add broad checked-in pixel baselines. |
| Source assertions | Add checks that the matrix imports or aliases `RulesteadAdmin.Components.*` and does not add Storybook, PhoenixStorybook, public admin router routes, release/publish files, schemas, or broad visual-diff tooling. |
