## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host telemetry provider -> `rulestead` contract | Raw provider results can overstate health, omit scope, or leak provider-specific semantics into core logic |
| runtime context -> guardrail query | Ambient scope inference can silently query the wrong environment or tenant |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-49-01 | T | `Rulestead.Guardrails.Provider` seam | mitigate | Keep provider interaction behind one behavior-only boundary with explicit arguments and no built-in adapters. |
| T-49-02 | I | `SignalFact` normalization | mitigate | Collapse weak, stale, or unsupported inputs into bounded fail-closed vocabularies instead of implied health. |
| T-49-03 | E | query construction | mitigate | Require explicit environment and tenant fields in query normalization and prove `Context` propagation in tests. |
| T-49-10 | R | provenance/audit compatibility | mitigate | Reuse or bridge `Command.GovernanceSupport`, `AuditEvent`, and `Telemetry` normalization helpers so guardrail metadata fits the existing audit envelope. |
</threat_model>
