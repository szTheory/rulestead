## Advisor Research Summary

### Dynamic URL Source

Recommended `DEMO_FRONTEND_URL` as backend runtime/application config, with `http://localhost:3000` as fallback. Alternatives rejected:

- Deriving from `DEMO_FRONTEND_PORT`: simple for direct localhost, but cannot represent proxy hosts or full origins.
- Phoenix endpoint/request origin: idiomatic for backend self-links, but FleetDesk is a sibling frontend, not the Phoenix endpoint.
- Config provider/release overlay: valid for complex deployments, but overkill for a demo URL already supplied by env.

### Phoenix Surface Coverage

Recommended explicit controller/layout assigns or attrs for the configured URL. Alternatives rejected:

- Template helper: smaller diff, but hides runtime config access in render code.
- Global plug assign: centralized, but overbroad for two links.
- New reusable component: clean if the launcher expands, but premature for this gap.

### Browser Evidence Shape

Recommended targeted compose-backed Playwright evidence:

- Assert both FleetDesk links render the normalized live `DEMO_FRONTEND_URL`.
- Click through at least one launcher path and assert FleetDesk loads.
- Keep screenshots as supporting artifacts.
- Avoid all-screen baseline expansion.

### Scope Boundary

Recommended narrow gap closure:

- Include runtime URL plumbing, Phoenix link rendering, copy correction for stale "port 3000" wording, targeted Playwright/backend tests, and verification artifact backfill.
- Exclude brand rework, palette/logo/token changes, admin polish, public API/schema/config-contract changes, release workflow edits, component framework work, `rulestead_admin` publish prep, and unrelated formatting cleanup.
