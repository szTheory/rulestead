<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Publish/mutation commands -> validator | Untrusted authored changes must be validated before durable publish/apply actions. |
| Validator findings -> adapter decisions | Incorrect blocker handling could permit broken dependency snapshots. |
| Blocked outcomes -> audit/support evidence | Findings must remain support-usable while avoiding sensitive leakage. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-54-07 | Tampering | Publish path | mitigate | Enforce validator gate before any publish side effect; abort on blockers. |
| T-54-08 | Elevation of privilege | Cross-tenant mutation apply | mitigate | Validate tenant scope for each dependency finding and block mismatches. |
| T-54-09 | Repudiation | Blocked mutation traceability | mitigate | Persist blocker findings and reason codes in support-safe audit metadata. |
| T-54-10 | Information Disclosure | Blocker payload overexposure | mitigate | Reuse redaction controls from Plan 01 and avoid raw trait/sample data. |
| T-54-11 | Denial of Service | Repeated expensive validation scans | mitigate | Feed validator from projection-backed inventory entries and bounded query scopes. |
| T-54-12 | Tampering | Adapter divergence (Ecto vs Fake) | mitigate | Enforce parity via shared contract tests and identical finding schemas. |
</threat_model>
