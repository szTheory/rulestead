# Plan 62-01: Schedule Hook And Idempotency Contract

<objective>
Register observation-window close ticks via `schedule_governed_action/1` inside `advance_rollout` (Fake + Ecto parity) with deterministic idempotency and supersession per CONTEXT D-01, D-02, D-03.
</objective>

<threat_model>
| Threat | Severity | Mitigation |
|--------|----------|------------|
| T-62-01: Duplicate ticks double-advance later | high | Deterministic idempotency_key + fetch-existing on conflict |
| T-62-02: Schedule failure rolls back advance | high | Post-transaction hook; advance returns {:ok, ...} even if schedule fails |
| T-62-03: Stale pending tick after manual re-advance | medium | cancel_superseded_auto_advance_ticks!/4 before insert |
| T-62-04: Non-deterministic correlation_id keys | medium | Explicit idempotency_key field bypasses correlation_id default |
</threat_model>

<task id="62-01-02" type="execute">
  <name>Auto-advance schedule hook in Ecto advance_rollout</name>
  <read_first>

  <action>
After `Repo.transact()` success in `advance_rollout/1` (line ~894 `{:ok, %{decision: decision, ruleset: ruleset}}`), call private `maybe_schedule_auto_advance_tick/2`:

**maybe_schedule_auto_advance_tick(command, %{decision: decision, ruleset: ruleset})**

1. `fetch_rollout_auto_advance_policy/1` for `(command.flag_key, command.environment_key, command.rule_key)`
2. If `{:error, _}` or policy `enabled: false` → `:ok` (no-op)
3. If policy enabled but incomplete fields → `:ok` (fail-closed, no schedule)
4. Require `command.monitoring_window_ends_at` — if nil, skip
5. Build idempotency_key:
   `"scheduled_execution:auto_advance:#{flag_key}:#{env_key}:#{rule_key}:#{command.stage}:#{DateTime.to_iso8601(command.monitoring_window_ends_at)}"`
6. Build command_snapshot:
```elixir
%{
  "rollout" => %{
    "rule_key" => command.rule_key,
    "stage" => command.stage,
    "percentage" => command.percentage,
    "monitoring_window_started_at" => command.monitoring_window_started_at,
    "monitoring_window_ends_at" => command.monitoring_window_ends_at
  },
  "auto_advance" => %{
    "policy_next_stage" => policy.next_stage,
    "policy_next_percentage" => policy.next_percentage,
    "observation_window_seconds" => policy.observation_window_seconds
  },
  "signal_facts" => []
}
```
7. Call `cancel_superseded_auto_advance_ticks!/4` for same `(flag, env, rule)` with state `"scheduled"` and metadata source guardrail_automation
8. `schedule_governed_action/1` with:
   - `action: :advance_rollout`
   - `scheduled_for: command.monitoring_window_ends_at`
   - `execution_mode: :policy_bypass`
   - `resource_type: "flag"`, `resource_key: command.flag_key`
   - `actor: %{"id" => "system:scheduler", "type" => "system", "display" => "Scheduler"}`
   - `metadata: %{"source" => "guardrail_automation", "automation_phase" => "evaluate_and_advance"}`
   - `idempotency_key: idempotency_key`
   - `reason: "auto_advance observation window close"`

9. On schedule error: emit telemetry/log, still return advance `{:ok, ...}` — do NOT change advance result

System scheduler actor string must match existing scheduled execution tests.
