# Phase 51 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | Phoenix LiveView `Phoenix.Component` function components; no new JS or CSS framework |
| Icon library | none for this phase; use text labels and existing `rs-*` semantic classes |
| Font | Inter for body, Sora for page titles if already loaded by host chrome, IBM Plex Mono or system monospace for keys and evidence values |

Source: `51-RESEARCH.md` standard stack, existing `rulestead_admin` component inventory, `prompts/rulestead-admin-ux-and-operator-ia.md`, and absence of `components.json`.

Phase 51 must extend the existing mounted `/flags/:key/rollouts` LiveView. Do not introduce a standalone guardrail dashboard, fleet observability page, charting dependency, browser-side metrics polling, or provider-specific panel.

---

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Badge internals, inline code gaps, compact metadata separators |
| sm | 8px | Guardrail fact row gaps, button groups, badge-to-label spacing |
| md | 16px | Default card padding, form field spacing, timeline row internals |
| lg | 24px | Main/aside grid gap, section padding, status panel padding |
| xl | 32px | Rollout page layout gap between workflow and sidebar |
| 2xl | 48px | Major break between status summary and intervention timeline |
| 3xl | 64px | Page-level vertical spacing only; avoid inside cards |

Exceptions: action targets must be at least 44px tall; compact badges may use 4px vertical padding only when paired with 8px horizontal padding; existing `rs-card` radius remains max 8px.

Layout contract:
- Keep the current two-column `rs-rollouts__layout`: primary workflow left, guardrail/status context right or directly below the rollout percentage card on narrow screens.
- Add guardrail status as one `rs-card` or dedicated `RolloutComponents.guardrail_status/1` section, not nested cards.
- Timeline excerpt rows remain separate repeated items using the existing audit row/card pattern.
- Primary focal point is the guardrail status panel state label and evidence summary; secondary priority is the intervention timeline excerpt.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 14px | 600 | 1.4 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px | 600 | 1.2 |

Use exactly these four sizes in new Phase 51 UI. Use `14px` for badges, table-like fact labels, timestamps, and timeline metadata. Use `16px` for explanations. Use `20px` for card headings. Use `28px` only for page-level title if this phase touches the shell heading.

Code-like values such as flag key, rule key, threshold operator, threshold value, sample counts, and correlation ids use monospace at the same role size as surrounding text.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#FAFAFA` | Mounted page background and main workflow surface |
| Secondary (30%) | `#F4F4F5` | Cards, status rows, sidebar blocks, timeline row backgrounds |
| Accent (10%) | `#EA580C` | Current live exposure stat, selected rollout ladder step, primary action focus/hover, warning attention for held or pending guardrail state |
| Destructive | `#DC2626` | Confirmed rollback-triggered state, destructive rollback labels, failed guardrail breach copy only |

Accent reserved for: current live exposure stat, selected ladder step, primary action focus/hover, `pending_data` and `held` attention labels, and the left border or badge on the guardrail status panel. Do not use accent for every link or every button.

Semantic state contract:
- `healthy`: restrained positive tone; label "Healthy"; never use celebratory copy.
- `pending_data`: accent/warning tone; label "Pending data"; explain stale, missing, or insufficient evidence.
- `held`: accent/warning tone with stronger border; label "Held"; explain automation is blocked fail-closed.
- `rollback_triggered`: destructive tone; label "Rollback triggered"; show confirmed breach and stable snapshot target when available.
- missing status row: neutral/warning tone; label "No guardrail decision recorded"; never render as green or "no issues".

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Preview sample |
| Empty state heading | No guardrail decision recorded |
| Empty state body | This rollout stage has guardrail definitions, but no evaluated decision has been recorded for this environment yet. Wire the host signal provider or run the guarded evaluation before treating the stage as healthy. |
| Error state | Guardrail status could not be loaded. Keep the rollout unchanged, then retry from this mounted page or review the per-flag audit timeline. |
| Destructive confirmation | Publish risky jump: Publish from `{current}%` to `{target}%` only after previewing the sample and recording a reason. |

Additional required copy:
- Status panel heading: `Guardrail status`.
- Threshold section label: `Thresholds and evidence`.
- Freshness label: `Freshness`.
- Sample label: `Sample`.
- Intervention timeline heading: `Guardrail interventions`.
- Automatic hold timeline title: `Automatic guardrail hold`.
- Automatic rollback timeline title: `Automatic guardrail rollback`.
- Manual action label: `Manual rollout action`.

State explanation copy:
- `healthy`: `Valid guardrail evidence is inside threshold for this rollout stage.`
- `pending_data`: `Automation is waiting for valid guardrail evidence and will not assume the stage is healthy.`
- `held`: `Guardrail automation held this rollout fail-closed. Review the missing or stale signal before advancing.`
- `rollback_triggered`: `A confirmed threshold breach triggered rollback to the last stable rollout snapshot.`
- unsupported or missing provider: `The host app has not provided a supported guardrail signal for this scope.`
- stale signal: `The latest signal is outside the configured freshness window.`
- insufficient sample: `The latest signal does not meet the configured minimum sample size.`

Do not use: "all clear", "safe", "dashboard", "monitoring", "fleet", "observability", "insights", or "automatic advance" in new Phase 51 UI copy.

## Interaction Contract

Guardrail status is read-only in Phase 51. It may explain and link; it must not evaluate signals, fetch provider metrics, trigger auto-advance, or mutate guardrail decisions from LiveView.

Required surfaces:
- Add a per-stage guardrail status panel to the existing rollout workflow, populated from `Rulestead.fetch_guardrail_status/3`.
- Show decision state, decision reason, effective percentage, monitoring window start/end, latest occurred time, freshness, threshold operator/value, observed value, and sample/minimum sample when present.
- Show authored guardrail definitions even when no decision row exists, so operators can see the expected host signal contract.
- Add a short intervention timeline excerpt on the rollout page using `Rulestead.list_audit_events/1`; include automatic hold/rollback rows and nearby manual rollout actions.
- Provide a link to the full per-flag timeline for complete audit history.

Manual versus automatic distinction:
- Automatic guardrail rows must include a visible `Automatic` label and system/source wording when metadata includes `actor_type=system` or `source=guardrail_automation`.
- Manual rollout rows must keep actor display prominent and must not share the automatic label.
- Automatic rollback must not be presented as generic audit rollback; copy must describe it as a fresh guarded rollout intervention from exact stable snapshot truth.

Missing and fail-closed behavior:
- A missing decision row renders the empty-state copy above.
- Stale, insufficient sample, missing provider, unsupported scope, and no stable rollback target render as blocking explanations, not as neutral absence.
- The UI must never hide the guardrail panel only because status is unavailable.

Accessibility contract:
- Guardrail status panel uses `aria-label="Guardrail status"`.
- Timeline excerpt uses `aria-label="Guardrail interventions"`.
- Status and error messages use existing `role="status"` and `role="alert"` patterns.
- Color cannot be the only state indicator; every status needs a text label.
- Buttons retain existing keyboard behavior and 44px target minimum.
