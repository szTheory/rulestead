<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| live compare -> saved plan artifact | Reviewed promotion preview state crosses into a durable applyable artifact here |
| saved plan artifact -> promote apply path | Previously generated plan data re-enters a mutating execution path here |
| CLI apply -> governed/protected target workflow | Automation-triggered promote apply crosses into existing approval and audit boundaries here |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-24-04-01 | T | saved plan artifact freshness | mitigate | Revalidate compare token, target fingerprint, and dependency closure before any apply, and map stale/fingerprint drift to domain rejection status `stale` or `blocked` per D-20 and D-26. |
| T-24-04-02 | E | protected-target CLI apply | mitigate | Route protected-target applies through the existing governed action path and never permit direct raw mutation from CLI automation per D-24. |
| T-24-04-03 | T | raw source/target apply surface | mitigate | Reject apply mode unless input is a previously generated saved plan artifact plus explicit `--reason`, preventing one-shot mutation from raw source/target pairs per D-10 and D-13. |
| T-24-04-04 | I | plan artifact contents | mitigate | Persist only bounded semantic plan data and reuse compare-derived metadata rather than serializing runtime snapshots, audit histories, drafts, or tenancy-expanding state. |
| T-24-04-05 | R | CLI mutation auditability | mitigate | Preserve compare token, fingerprints, bounded flag scope, and reason in the saved plan/apply contract so audit/governance surfaces can link an apply back to the reviewed preview artifact. |
</threat_model>
