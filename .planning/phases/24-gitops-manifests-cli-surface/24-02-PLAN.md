<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| manifest file -> parser/validator | User-provided manifest content enters the semantic validation path here |
| diff basis -> result envelope | Two authored-state inputs are transformed into change/blocker findings here |
| result envelope -> stdout/exit code | Domain status is translated into CLI output and process status here |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-24-02-01 | T | manifest parser and validator | mitigate | Validate kind/schema/version and authored dependency references before rendering success, and map invalid domain state to status `invalid` with exit code `3` per D-18 and D-19. |
| T-24-02-02 | R | output/render contract | mitigate | Build text and JSON from one canonical result envelope so automation and humans cannot receive conflicting interpretations of the same validation/diff outcome. |
| T-24-02-03 | I | JSON renderer stdout | mitigate | Keep `--format json` stdout pure JSON per D-16 and move warnings/details into the envelope instead of mixed prose. |
| T-24-02-04 | D | diff domain service | mitigate | Reuse compare-style finding taxonomy and stable ordering rather than recomputing multiple inconsistent diff views that could amplify review noise or produce unstable CI output. |
| T-24-02-05 | E | task-level exit mapping | mitigate | Reserve exit code `1` for usage/I/O/system failure and keep domain statuses on `0`, `2`, or `3` so callers cannot accidentally treat blocked/invalid previews as process crashes. |
</threat_model>
