<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| manifest -> saved plan artifact | Reviewed import preview state crosses into a durable applyable artifact here |
| saved plan artifact -> apply path | Previously generated plan data re-enters a mutating execution path here |
| adapter command -> governed/protected target workflow | Import apply crosses into existing approval and audit boundaries here when protection requires it |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-24-03-01 | T | saved plan artifact freshness | mitigate | Revalidate schema version, plan token, compare token where present, target fingerprint, and dependency closure before any apply, and map stale/fingerprint drift to domain rejection status `stale` or `blocked` per D-20 and D-26. |
| T-24-03-02 | E | protected-target CLI apply | mitigate | Route protected-target applies through the existing governed action path and never permit direct raw mutation from CLI automation per D-24. |
| T-24-03-03 | T | raw-manifest apply surface | mitigate | Reject apply mode unless input is a previously generated saved plan artifact plus explicit `--reason`, preventing one-shot mutation from raw manifests per D-10 and D-13. |
| T-24-03-04 | I | plan artifact contents | mitigate | Persist only bounded semantic plan data, not runtime snapshots, audit histories, drafts, or tenancy-expanding state; keep artifacts aligned to authored-state and apply review needs only. |
| T-24-03-05 | R | CLI mutation auditability | mitigate | Preserve compare token or manifest plan token, fingerprints, bounded flag scope, and reason in the saved plan/apply contract so audit/governance surfaces can link an apply back to the exact reviewed preview artifact. |
</threat_model>
