<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| backend compare payload -> mounted LiveViews | Canonical compare data crosses into operator-facing UI surfaces here |
| mounted session/policy envelope -> compare routes | Existing admin authz and environment state govern access to summary and drill-in compare screens here |
| progressive disclosure UI -> raw compare details | Lower-level machine data becomes available to operators on demand here |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-22-02-01 | E | compare routes | mitigate | Mount compare pages inside the existing `rulestead_admin` router macro and `Live.Session` on-mount policy flow so no separate auth or session path bypass appears. |
| T-22-02-02 | T | stale-preview interpretation | mitigate | Render hard-stale compare results as blocker banners with explicit stale reasons and token metadata, never as silently reusable previews. |
| T-22-02-03 | I | raw compare disclosure | mitigate | Keep machine payload and structured low-level fields behind progressive disclosure and render only the canonical authored compare data, not runtime internals or hidden session state. |
| T-22-02-04 | R | read-only phase boundary | mitigate | Exclude apply, schedule, governance-execution, and publish controls from compare routes so the UI cannot imply mutation happened or is available in Phase 22. |
</threat_model>
