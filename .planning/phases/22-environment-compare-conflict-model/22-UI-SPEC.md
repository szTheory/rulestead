# Phase 22 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | Phoenix.Component + Phoenix LiveView function components |
| Icon library | none locked; use inline SVG icons sized to the typography grid |
| Font | Sora headings, Inter body/labels, IBM Plex Mono for keys, compare tokens, versions, and structured diff fields |

Notes:
- Source: `rulestead_admin` is a Phoenix LiveView package, not a React app, so shadcn is not applicable.
- Source: the brand direction keeps the UI calm, infrastructure-grade, and explainable under operator pressure.
- Source: existing admin surfaces already use the `rs-*` component vocabulary, `Shell.page`, `OperatorComponents`, `AuditComponents`, and `FlagComponents`; extend those rather than introducing a parallel styling system.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline badge gaps, diff markers, token metadata spacing |
| sm | 8px | Compact row spacing, status chip gaps, finding count clusters |
| md | 16px | Default card padding, row internals, control spacing |
| lg | 24px | Section padding, context bar spacing, filter/action grouping |
| xl | 32px | Summary-to-list gaps, desktop two-column compare spacing |
| 2xl | 48px | Major transitions between overview, findings, and per-flag results |
| 3xl | 64px | Page-level vertical breathing room within host chrome |

Exceptions: 44px minimum hit area for icon-only disclosure toggles and row actions.

Accessibility note:
- Expand/collapse controls for raw payload and structured diffs must include an `aria-label` naming both the action and the flag key.
- Hairline dividers for card boundaries, grouped findings separators, and source/target/proposed columns are border/stroke rules, not spacing tokens.

|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 14px | 600 | 1.4 |
| Heading | 20px | 600 | 1.2 |

Additional contract:
- Use IBM Plex Mono at 14px / 400 / 1.4 for flag keys, compare tokens, ruleset versions, audience keys, machine payload snippets, and source-target field values where exactness matters.
- Reserve Display for page titles, overall compare status banners, and hard-stale blocker banners only.
- Keep per-flag rows and finding chips at Body + Label only; do not escalate routine drift into oversized typography.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#F3F6F8` | App background, compare canvas, summary page background, expanded diff background |
| Secondary (30%) | `#E3EBF1` | Cards, grouped findings buckets, env context rail, per-flag result rows |
| Accent (10%) | `#2F6FED` | Primary CTA, active compare row, current environment chip, disclosure affordance, focus ring |
| Destructive | `#C65A3A` | Hard-stale blockers, archived-target revive blockers, destructive/governed-action warnings only |

Accent reserved for: `Run compare` CTA, currently selected flag drill-in row, active environment/source-target locator chips, focus-visible rings, and progressive-disclosure controls for structured detail. Do not use accent for every link, every badge, or non-blocking warnings.

Additional semantic tones:
- Positive: `#2E7D5B` for safe/ready informational states such as `no blockers` and `in sync`.
- Warning: `#C28A2E` for applyable-with-risk findings such as operational overrides, protected-target notices, and unpublished draft presence.
- Neutral ink: `#22313D` for primary text; muted ink `#5E7282` for helper text, token metadata, and row subtitles.
- Production emphasis: keep the existing shell `data-env-tone` contract. When the target environment is production, add a thin destructive-tinted stripe and plain-language policy copy at the page header.

Source/default rationale:
- Source: prior admin UI specs already locked the mineral neutral + controlled blue + restrained ember system.
- Source: Phase 22 severity is derived from apply safety, so ember stays mapped to blocker/destructive semantics rather than becoming a generic accent.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Run compare |
| Empty state heading | No comparable differences found |
| Empty state body | Source and target match for the selected authored scope. Change the environment pair or open a flag to review exact published state. |
| Error state | We couldn't build a compare preview for this environment pair. Refresh the page, confirm both environments still exist, and retry after any pending publish finishes. |
| Destructive confirmation | Future apply handoff only: "This preview is read-only in Phase 22. Review blockers, warnings, and the compare token before opening governed apply in a later phase." |

Additional wording rules:
- Always use explicit directionality: `source`, `current target`, `proposed target after apply`.
- Use `published`, `draft`, `dependency`, `drift`, `conflict`, `compare token`, and `operational override` exactly; do not collapse them into vague terms like `change set` or `delta` in operator-facing copy.
- Findings copy must name the consequence first, then the next step: `Missing audience dependency. Publish or restore the source dependency before apply.`
- Warning copy must never imply mutation already happened. Use `would`, `after apply`, and `if promoted`.
- Draft presence copy should read as unpublished work, not as part of the current compare basis.

## Interaction Contract

### Route-backed surfaces

- Add a dedicated read-only environment summary route for compare discovery and a per-flag compare drill-in route for structured authored-state review.
- Keep source and target environment selection explicit and URL-backed; do not introduce hidden session-only compare state.
- Preserve the existing `?env=` discipline by keeping the mounted admin’s current environment explicit while also carrying compare-specific source/target params in the URL.

### Screen hierarchy

- Summary page order: context bar, overall status banner, count summary, findings buckets, per-flag result rows, then progressive disclosure for raw payload.
- Per-flag drill-in order: context bar, flag status summary, findings grouped by severity, three-column or stacked source/current-target/proposed-target sections, then structured diff disclosure.
- Raw machine payload is never above the fold.

### Compare posture

- Phase 22 is preview-only. No apply, schedule, bulk promotion, or destructive action controls appear on the compare summary route.
- The summary page lists only differing or problematic flags by default.
- Every finding is rendered with both severity and typed class.
- Severity color follows apply safety, not visual magnitude of diff.

### Findings and disclosure

- Findings buckets appear in strict order: blockers, warnings, info.
- Each bucket shows count, one-sentence meaning, and the operator’s next likely action.
- Per-flag rows show the smallest useful preview: severity summary, changed field count, dependency count, drift/conflict count, and explicit source-target direction.
- Structured field diffs and raw payload are collapsed behind `details`-style disclosure or an equivalent accessible progressive-disclosure component.

### Staleness and token behavior

- Show the compare token in the drill-in and summary metadata rail, but behind secondary emphasis.
- Hard-stale compare results render as blocker state with the banner copy first and the stale reason second.
- Age-based warning badges may appear after the short warning window, but elapsed time alone must never present as expiry.
- Unrelated environment churn must not invalidate the visible compare result unless the canonical token says the scoped authored set changed.

### Environment and policy signaling

- The target environment owns the risk framing because apply safety is target-oriented.
- Production-target compares show destructive-tinted shell emphasis plus plain-language copy: `Production target. Review blockers and governed-apply requirements before continuing.`
- Protected-target requirements and operational overrides are warnings, visually distinct from blockers.
- Draft rulesets in the source are rendered as unpublished work banners, not as authored compare diffs.

### Feedback and async behavior

- Summary compare loads may use connected-only async loading if the result spans many flags.
- Loading copy must keep operator orientation: `Building authored compare for staging -> prod.`
- Failure copy must stay scoped and actionable; do not suggest peer-health or runtime-debug remedies unless the compare payload explicitly indicates that issue.

### Mobile behavior

- On narrow screens, stack the source/current-target/proposed-target sections vertically in that order.
- Per-flag result rows collapse to: flag key, highest severity, changed fields count, and an expand action.
- Findings buckets remain above the result list; raw payload stays hidden behind explicit disclosure.

## Component Inventory

Required shared components for Phase 22:

| Component | Purpose | Notes |
|----------|---------|-------|
| `rs-banner` | Overall compare status, hard-stale notice, unpublished draft notice | Extend existing `OperatorComponents.banner/1` tones and copy posture |
| `rs-summary-grid` | Counts for blockers, warnings, info, changed flags, dependency gaps | Reuse `OperatorComponents.summary_grid/1` and existing `rs-stat` cards |
| `rs-status-list` | Findings buckets and target policy state summaries | Reuse `OperatorComponents.status_list/1` |
| `rs-trace-panel` | Compare token metadata, authored heads/fingerprints, scoped environment facts | Reuse `OperatorComponents.trace_panel/1` with mono values |
| `rs-diff-card` | Readable source/current-target/proposed-target slices | Extend existing diff-card pattern for explicit directionality |
| `rs-policy-state` | Production-target and governed-apply warning copy | Reuse existing policy component and target-oriented wording |
| `rs-card` / `section_card` | Surface wrapper for overview, grouped findings, and flag drill-in sections | Keep the current shell/card contract unchanged |

Extend existing components instead of replacing them:
- `Shell.page` remains the shared mounted page frame and environment emphasis source.
- `FlagComponents.section_card/1` remains the default compare surface wrapper.
- `AuditComponents.diff_card/1` is the preferred base for readable field-level compare rendering.
