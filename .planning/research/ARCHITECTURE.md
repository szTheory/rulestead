## Architectural Thesis

The brand book already exists as `prompts/rulestead-brand-book.md`. v1.14 promotes it to a machine-assisted source of truth by extracting the palette and token semantics into a new `brandbook/` tree, then re-skinning the shipped admin CSS to match — with zero build-time coupling. No toolchain changes affect the auto-publish pipeline.

The entire architecture mirrors the pattern already proven by `check_synced_pair.py`: drift is detected by a comparison script run in CI, not enforced by a preprocessor or build dependency. The brand system stays inert at runtime; it only influences the shipped CSS through deliberate, human-authored edits that are then validated by CI.

### Rationale

- All brand collateral lives under one tree with no build tooling dependency. Consuming it means reading the files; the repo has no preprocessor step.
- `tokens.json` is the machine-readable canonical. `tokens.css` is a hand-authored CSS shadow of its colour section — kept because it lets a phase planner paste-test tokens against the cascade without running any tooling.
- `assets/logo/` keeps only SVGs. No PNG exports committed (size budget concern); PNGs are generated on demand from SVG by any consumer.
- `specimens/` are pure SVG documents — viewable in a browser, diffable in git, zero binary blobs.

## Source-of-Truth + Sync Model

```
tokens.json         "stead-blue": #3A6F8F
      ↕ (human edit, then run check)
rulestead_admin.css  Block 1: --rs-primary: #3A6F8F;
                     Block 4: --rs-primary: #3A6F8F;   ← synced-pair light
                     Block 2: --rs-primary: [dark-adjusted hex];
                     Block 3: --rs-primary: [dark-adjusted hex];  ← synced-pair dark
```

**No build step converts tokens.json into CSS.** The conversion is performed once by a human during the Phase 97 re-skin, and thereafter kept honest by `check_brand_tokens.py`. This mirrors exactly how `check_synced_pair.py` works: it detects drift but does not auto-correct.

The "re-skin scope" is colour tokens only — specifically the brand/accent/neutral-ramp values in Blocks 1–4. The invariant tokens (typography, spacing, radius, z-index, motion) in `:root` are untouched.

## Drift-Check Design: `scripts/check_brand_tokens.py`

Modelled directly on `check_synced_pair.py`. The approach:

1. Parse `brandbook/tokens.json` → build a dict of `{ token_name: hex_value }` from `admin_css_mapping.light` resolved through `palette`.
2. Parse `rulestead_admin.css` (strip comments first, same as existing script) → extract `--rs-primary`, `--rs-accent`, and the neutral ramp hex literals from Block 1 (`.rs-shell, [data-rulestead]` — the light default).
3. Compare: for each mapped token, assert the hex literal in Block 1 matches the brand palette value.
4. Exit 0 + print `BRAND TOKENS SYNCED (N tokens)` on success; exit 1 with per-token diff on mismatch.

```python
CSS    = "rulestead_admin/priv/static/css/rulestead_admin.css"

def light_block_decls(css):
    # Find ".rs-shell," selector (Block 1), extract --rs-* declarations
    # Same decls() approach as check_synced_pair.py
    ...

def main():
    import json, re, sys
    palette = json.load(open(TOKENS))["palette"]
    mapping = json.load(open(TOKENS))["admin_css_mapping"]["light"]
    raw = open(CSS).read()
    css = re.sub(r"/\*.*?\*/", "", raw, flags=re.S)
    live = light_block_decls(css)   # dict of --rs-name: hex_value
    mismatches = []
    for token, palette_name in mapping.items():
        expected = palette[palette_name]["hex"]
        actual   = live.get(token)
        if actual and actual.lower() != expected.lower():
            mismatches.append((token, expected, actual))
    if not mismatches:
        print(f"BRAND TOKENS SYNCED ({len(mapping)} tokens)")
        return 0
    for (t, exp, act) in mismatches:
        print(f"  DRIFT  {t}: expected {exp}, found {act}")
    return 1
```

**Why only Block 1?** Block 1 is the light default and the stable canonical. If Block 1 matches tokens.json and `check_synced_pair.py` passes (Blocks 1+4 identical, Blocks 2+3 identical), then all four blocks are consistent with intent. The two scripts together form a complete invariant.

## SVG Size Budget CI Lint

Add to `scripts/ci/lint.sh` (MODIFIED — append, do not rewrite):

```bash
# Brand asset size budget
MAX_LOGO_BYTES=20480   # 20 KB per SVG in brandbook/assets/logo/
MAX_SPECIMEN_BYTES=51200  # 50 KB per SVG in brandbook/assets/specimens/

fail=0
for f in brandbook/assets/logo/*.svg; do
  size=$(wc -c < "$f")
  if [ "$size" -gt "$MAX_LOGO_BYTES" ]; then
    echo "SIZE BUDGET EXCEEDED: $f ($size bytes > $MAX_LOGO_BYTES)"
    fail=1
  fi
done
for f in brandbook/assets/specimens/*.svg; do
  size=$(wc -c < "$f")
  if [ "$size" -gt "$MAX_SPECIMEN_BYTES" ]; then
    echo "SIZE BUDGET EXCEEDED: $f ($size bytes > $MAX_SPECIMEN_BYTES)"
    fail=1
  fi
done
[ $fail -eq 0 ] && echo "SVG SIZE BUDGET OK"
exit $fail
```

These limits keep the repo lean. A well-authored SVG wordmark should be 3–8 KB; 20 KB gives generous headroom for path complexity. Specimens at 50 KB per file are generous enough for multi-element documents.

## Architectural Patterns

### Pattern 1: Mirror-Not-Generate

**What:** `tokens.json` is the canonical palette record; `tokens.css` and the admin CSS are hand-authored mirrors of its colour values, not generated outputs.

**When to use:** When you want zero build-time coupling and a repo that publishes to Hex without any pre-build step.

**Trade-offs:** Drift is possible; mitigated entirely by `check_brand_tokens.py`. The upside is that the auto-publish pipeline has no new dependency — merging to `main` publishes exactly as before.

### Pattern 2: Additive Drift Check

**What:** New CI checks are appended to the existing `scripts/ci/lint.sh` rather than creating new workflow files or CI steps.

**When to use:** When the existing CI surface is scripts-first and adding a new workflow file would be heavier than necessary.

**Trade-offs:** `lint.sh` grows longer; acceptable because each check is a single `python3` invocation with a clear exit code.

### Pattern 3: Synced-Pair Coverage by Composition

**What:** `check_synced_pair.py` already guarantees Blocks 2+3 = dark and (implicitly, by structure) Blocks 1+4 = light are identical to each other. `check_brand_tokens.py` guarantees Block 1 matches `tokens.json`. Together, if both pass, all four blocks are consistent with brand intent.

**When to use:** Always — run both scripts after any colour token edit.

**Trade-offs:** Two separate scripts rather than one; this is a feature, not a bug. Each script has a single responsibility and can be explained, debugged, or replaced independently.

---

## Anti-Patterns

### Anti-Pattern 1: Preprocessor Coupling

**What people do:** Add SCSS/PostCSS/style-dictionary that generates `rulestead_admin.css` from `tokens.json` at build time.

**Why it's wrong:** The auto-publish pipeline merges to `main` and publishes both Hex packages. Adding a build step introduces a new failure mode with no rollback. The shipped CSS becomes a derived artifact instead of a committed, reviewable file.

**Do this instead:** Hand-mirror the colour values and use `check_brand_tokens.py` to detect drift at CI time.

### Anti-Pattern 2: Committing PNG/Binary Brand Exports

**What people do:** Export PNG renders of all logo variants and commit them alongside SVGs.

**Why it's wrong:** Binary blobs inflate git history permanently, are not diffable, and add repo size that every `git clone` of the library pays for.

**Do this instead:** Commit only SVGs. Document in `brand-usage.md` how to rasterise locally with `rsvg-convert` or Inkscape.

### Anti-Pattern 3: Editing Only One Block of a Synced Pair

**What people do:** Update `--rs-primary` in Block 1 but forget Block 4 (or vice versa).

**Why it's wrong:** `check_synced_pair.py` catches this and fails CI, but only if you run it. It is easy to miss in a PR if the reviewer does not check CI output.

**Do this instead:** Always edit the light blocks (1+4) as a unit in one commit, then the dark blocks (2+3) as a unit. Treat the pair as a single atomic change.

### Anti-Pattern 4: Scattering Brand Collateral

**What people do:** Put the brand mark in `rulestead_admin/priv/static/`, the tokens in `guides/`, and the brand book stays in `prompts/`.

**Why it's wrong:** There is no single place to send a new contributor or a future design-system consumer. Drift becomes silent because there is no clear canonical.

**Do this instead:** Keep everything brand-collateral under `brandbook/`. Other locations hold only copies with a pointer comment.
