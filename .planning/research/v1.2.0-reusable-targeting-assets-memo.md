# v1.2.0 Candidate Memo: Reusable Targeting Assets

**Project:** Rulestead  
**Candidate:** `v1.2.0 — Reusable Targeting Assets`  
**Date:** 2026-05-23  
**Overall assessment:** Medium-value, high-leverage later milestone; not the best immediate next milestone.

## Executive Take

Reusable targeting assets are a real platform maturity step. The ecosystem has converged on them because repeated targeting logic becomes expensive, inconsistent, and hard to audit once teams operate many flags. Unleash segments, LaunchDarkly segments, and vendor rollout templates all validate the same need: define once, reuse many times.

For Rulestead right now, this is still probably **candidate 3, not candidate 1**. The current planning surface already says lifecycle and ownership remain the strongest follow-on gap after tenancy shipped, and the JTBD map says operators are still missing an ownership-heavy workbench and stale-state triage more urgently than reusable rule assets. Reusable targeting helps scale authored rules; lifecycle/ownership helps teams keep the product trustworthy every day.

The right way to think about this milestone is not "add segments because competitors have segments." The right question is whether shared targeting reduces real operator pain **without widening the product shape**. It can, if the milestone stays disciplined: shared audiences first, limited rollout templates second, no standalone control plane drift, no hidden precedence system, no cross-package leakage.

## Pros, Cons, Tradeoffs

### Pros

- Reduces repeated rule authoring across many flags, especially entitlement, geography, beta-program, and internal-user targeting.
- Improves operator consistency: one fix to a shared asset updates all dependent flags.
- Strengthens auditability because repeated targeting logic stops being copied-and-mutated ad hoc.
- Fits Rulestead's existing canonical language: user-facing `Audience`, internal matching segment semantics.
- Maps cleanly to the mounted admin model: list/detail/reference counts, "used by" screens, preview, and change audit.

### Cons

- Increases blast radius. Editing one shared asset can change many flags at once.
- Adds another layer of indirection to explainability: "why did this flag resolve this way?" becomes "rule matched because shared audience X matched."
- Raises diff/promotion/import complexity because dependencies now include reusable assets.
- Risks product-shape drift if "reusable targeting assets" expands into a generic template/workflow system.
- Can distract from the more immediate post-GA pain: ownership, stale cleanup, and lifecycle hygiene.

### Tradeoffs

- **Shared audiences first vs broad templates first:** audiences are idiomatic and bounded; templates are more powerful but more product surface.
- **Reference by stable key vs inline snapshot copy:** stable key is operator-friendly and reusable; snapshot copy is safer for promotions but defeats reuse. Rulestead should prefer stable keys plus validation.
- **Immediate propagation vs explicit preview:** immediate propagation is simpler; explicit impact preview is safer. Safety should win.

## What This Looks Like In Product Terms

### Library / runtime

- Keep evaluation shape the same: flags still evaluate ordered rules against explicit context.
- Add a first-class reusable asset reference inside rules, for example an audience condition that points at `audience_key: "enterprise_accounts"`.
- Keep targeting deterministic and local. Runtime snapshots should compile referenced assets into the same immutable snapshot version used for evaluation.
- Explain traces should name both the matched rule and the reused asset, for example: `ruleset v12 -> rule 2 matched -> audience enterprise_accounts matched -> variant treatment`.

### Admin UX

- `Audiences` becomes a stronger primary surface, not just a side object.
- Audience detail needs:
  - definition
  - description
  - owner
  - "used by N flags"
  - preview/sample membership
  - impact preview before save
- Flag rule editor should let operators pick an existing audience instead of recreating conditions.
- Editing a shared audience should show a "this affects X flags in Y environments" preview before confirm.

### Limited template UX

- If templates ship at all, keep them narrow: rollout/targeting presets for common patterns such as:
  - `internal_only`
  - `beta_users_then_10_percent_then_50_percent`
  - `enterprise_entitlement`
- Templates should create draft rules, not magical live inheritance.
- Do not ship a generalized workflow engine or automated release-plan system in this milestone.

## What Is Idiomatic For Elixir / Plug / Ecto / Phoenix

### Elixir / Phoenix

- Mounted companion admin remains idiomatic. `Phoenix.Router.forward/4` explicitly supports mounting a library-owned router under a host-owned prefix, which matches Rulestead's sibling-package model.
- Keep web concerns in `rulestead_admin`; keep domain logic in `rulestead`. LiveViews should orchestrate, not own targeting logic.
- Use URL-param-scoped tenant/environment context consistently in admin flows, matching the existing admin IA.

### Ecto

- Treat shared-asset edits as authored-state mutations with one audit narrative and dependency validation inside a transaction.
- `Ecto.Multi` is the right primitive for "update audience, validate impacted rules, write audit row, enqueue snapshot publish" flows.
- Store stable authored keys and explicit join/reference tables; do not encode reusable targeting as opaque JSON blobs if the admin UI must diff and audit it.

### Plug / host seam

- Keep host-auth and host-policy ownership intact. Shared targeting does not justify new auth assumptions.
- Preserve the current explicit-context seam. Do not make runtime implicitly query admin-only metadata or host data sources during evaluation.

### Package-boundary discipline

- `rulestead` owns schemas, validation, snapshot compilation, evaluation, explain traces, import/export semantics.
- `rulestead_admin` owns authoring screens, previews, and operator copy.
- No admin-only concepts should leak into the core public runtime API unless needed for authored-state references or explainability.

## Lessons From Comparable Systems

### Unleash

What it did right:

- Segments are a reusable collection of constraints, explicitly positioned as a way to avoid repeating rules.
- Release templates standardize rollout shape across flags.
- Lifecycle metadata and stale-state remain first-class, which is the more important everyday hygiene loop.

What to learn:

- Reuse works best when it is constrained and named clearly.
- Templates are useful, but they sit closer to release management than to basic flagging.

What to avoid:

- Letting reusable assets become an excuse to skip lifecycle/ownership work. Unleash's own model keeps both.

### LaunchDarkly

What it did right:

- Segments are an explicit reusable targeting primitive.
- The docs are clear that this reduces rebuilding rules flag-by-flag.
- Their product also reinforces that segments are environment-scoped, which prevents false assumptions.

What to learn:

- Shared targeting needs clear scoping rules and dependency visibility.
- Operators need to know what flags consume a shared asset before changing it.

What to avoid:

- Environment-specific objects with the same human name becoming a surprise footgun in compare/promotion flows.

### GrowthBook

What it did right:

- Strong emphasis on local evaluation, explainability, and operator comprehension.
- Draft/publish/review posture is calmer than "edit live and hope."

What to learn:

- If Rulestead ships templates, they should draft concrete rules rather than introduce hidden inheritance semantics.
- Shared logic is only safe when users can still understand the final evaluated rule path.

What to avoid:

- Reuse primitives that make explainability harder than the saved authoring time is worth.

### FunWithFlags

What it did right:

- Strong embedded runtime posture: fast local reads, host-mounted UI, clear separation between runtime and dashboard.

What to learn:

- Rulestead should keep the same embedded discipline: reusable targeting must not turn evaluation into remote control-plane dependency.

What to avoid:

- Any model that weakens deterministic local evaluation or introduces surprising precedence.

## Footguns Rulestead Should Avoid

- Editing one shared audience and silently changing dozens of flags without impact preview.
- Introducing hidden precedence between inline conditions and reusable assets.
- Reusing environment-local assets across promotion/import flows without explicit validation.
- Shipping templates that behave like ongoing inheritance instead of draft generation.
- Allowing "all tenants" shared targeting shortcuts that violate the current explicit, fail-closed tenancy posture.
- Storing admin-only template semantics in a way that complicates runtime snapshots or public API guarantees.

## DX And Operator UX Guidance

- Principle of least surprise: the final evaluated rule path must still be easy to explain.
- Host-app friendliness: no new required background services, no remote evaluation dependency, no host-auth expansion.
- Operator UX: every shared-asset mutation needs preview, reference counts, environment/tenant scope, and audit reason.
- Contributor DX: stable internal boundaries matter more than feature breadth; keep reusable-asset compilation as a snapshot concern, not a LiveView concern.

## Requirement Themes If This Becomes The Milestone

### Good themes

- Shared audiences as first-class reusable targeting assets.
- Safe authoring and safe mutation previews for reused assets.
- Explainability carry-through for reused targeting.
- Promotion/import/export dependency validation for reused assets.
- Limited preset generation for common targeting patterns, only if it stays draft-based.

### Non-goals

- Standalone control-plane drift.
- Generic workflow/release orchestration templates.
- Cross-environment hidden inheritance.
- Tenant-partitioned authored storage.
- Arbitrary org/project taxonomy expansion.

## Architectural Guardrails

- One canonical authored asset key, explicit references, and immutable audit history.
- Snapshot compiler resolves reusable assets into the runtime snapshot so evaluation stays local and deterministic.
- Shared-asset mutation path must be transactional and dependency-aware.
- Promotion/import must fail closed when referenced audiences/templates are missing or incompatible.
- Explain traces must surface reused asset involvement explicitly.
- Template support, if any, must generate draft rulesets or rollout drafts; it must not create an always-live inheritance graph.

## Recommendation

**Defer this as the immediate next milestone.** Keep it as a strong follow-on candidate after lifecycle/ownership work.

Rationale:

- The current project docs already rank lifecycle and ownership as the stronger post-GA gap.
- The JTBD map shows ownership/stale-state triage is still missing across builder, lead, and operator personas.
- Reusable targeting assets are valuable once rule volume is high, but lifecycle hygiene improves trust and operability even at modest scale.
- Segments/templates increase dependency and blast-radius complexity; Rulestead should add that after first making ownership, stale-state, and cleanup unmistakably first-class.

If this milestone is pulled forward anyway, scope it to **shared audiences first** and treat template work as optional and sharply bounded.

## Guardrails

- Build `Audience` reuse before any broader "template" system.
- Do not add hidden inheritance or live-linked rollout plans.
- Keep evaluation local; reusable assets compile into snapshots.
- Require impact preview, reference counts, and audit reason for shared-asset edits.
- Preserve explicit tenant and environment scope everywhere.
- Keep `rulestead` as the domain/runtime package and `rulestead_admin` as the mounted authoring surface.
- Do not widen into standalone admin, project/workspace taxonomy, or enterprise workflow automation in this milestone.
