# Milestone Candidate Memo: v1.2.0 Guarded Rollout Foundations

**Project:** Rulestead  
**Date:** 2026-05-23  
**Confidence:** Medium-high

## Context

Rulestead has already shipped deterministic local evaluation, rollout controls, audit/governance, scheduling, diagnostics, promotion, GitOps, RBAC, and bounded tenancy. The current planning docs still identify lifecycle and ownership as the strongest next milestone candidate, while guarded rollout is explicitly listed as valuable but later because it is a larger product jump that needs metric inputs, thresholds, automated reactions, and stronger operator UX. See [.planning/PROJECT.md](/Users/jon/projects/rulestead/.planning/PROJECT.md:77), [.planning/ROADMAP.md](/Users/jon/projects/rulestead/.planning/ROADMAP.md:9), and [.planning/research/JTBD-MAP.md](/Users/jon/projects/rulestead/.planning/research/JTBD-MAP.md:139).

## What This Milestone Would Mean

In Rulestead terms, "guarded rollout foundations" should not mean "full LaunchDarkly clone." It should mean:

- Operators can attach one or more host-supplied health signals to an existing staged rollout.
- Each rollout stage can pause or roll back when a guardrail crosses a configured threshold.
- The runtime package stays metric-source agnostic; the host app provides metrics and alert facts.
- The admin package surfaces status, reasons, and audit evidence inside the existing mounted rollout workflow.

That shape is consistent with Rulestead's existing split: runtime stays local and deterministic, admin stays mounted and host-owned, and high-blast-radius actions remain previewed, governed, and auditable.

## Pros

- Strong safety story for Tech Lead and SRE personas. The current JTBD gap is real: Rulestead has rollout controls but no metric-linked progressive rollback loop. See [.planning/research/JTBD-MAP.md](/Users/jon/projects/rulestead/.planning/research/JTBD-MAP.md:162).
- Clear market signal. LaunchDarkly and GrowthBook both treat monitored rollout as a serious maturity feature rather than a cosmetic progress bar.
- Reuses existing surfaces instead of inventing a new product area. The admin IA already has rollout stages, health sidebars, confirmations, scheduling, and audit-friendly modal flows. See [prompts/rulestead-admin-ux-and-operator-ia.md](/Users/jon/projects/rulestead/prompts/rulestead-admin-ux-and-operator-ia.md:175).
- Good operator value per feature. Even a narrow first version can be useful if it only supports "pause" and "rollback to previous stable state" from a small set of host metrics.

## Cons

- It is a control-plane complexity jump, not a small polish milestone. Rulestead would need a durable metric contract, evaluation windows, threshold semantics, rollback behavior, alerting copy, and incident-grade audit trails.
- It creates footgun risk if shipped half-formed. False positives, noisy metrics, missing exposure volume, or ambiguous thresholds will make operators trust it less than manual rollout.
- It competes with a more immediate post-GA gap. Lifecycle and ownership are still the more everyday pain reliever for builders, leads, and operators, and they fit the current architecture more directly.
- It can accidentally widen `rulestead_admin` into "org control plane" territory if the milestone chases dashboards, anomaly engines, or fleet health instead of bounded rollout safety.

## Tradeoffs

- **Differentiation vs focus:** Guarded rollout is more differentiating than lifecycle hygiene, but less aligned with the current "smallest coherent next move" principle.
- **Operator value vs host-app burden:** A bounded host-metrics seam preserves package discipline, but it also means adopters must already have usable telemetry and alert signals.
- **Automation vs principle of least surprise:** Automatic rollback is powerful only when it is predictable. "Hold and page" is often a better foundation than "flip traffic automatically on weak evidence."
- **Safety vs explainability:** Every automated stage decision must be inspectable in the same way a flag evaluation is inspectable.

## Product / Library / Admin Examples

### Product

- A `checkout_v2` rollout is staged at `5% -> 25% -> 50% -> 100%`.
- The operator attaches two guardrails: `checkout_error_rate` and `p99_checkout_latency`.
- When the rollout advances to `25%`, Rulestead starts a monitoring window for that stage only.
- If `checkout_error_rate` exceeds `1.5x baseline` for `10m`, the rollout enters `held` or `rolled_back`, and the timeline records the exact metric snapshot and threshold breach.

### Library

- `rulestead` exposes a host seam like `Rulestead.RolloutSignals` or `Rulestead.RolloutGuardrails`.
- The seam returns normalized facts, for example:
  - current value
  - baseline value
  - sample size
  - evaluation window
  - freshness timestamp
  - confidence status such as `:ok | :insufficient_data | :stale_signal | :breached`
- Core rollout execution consumes those normalized facts but does not fetch Prometheus, StatsD, or OpenTelemetry backends directly.
- Rollback remains a governed mutation executed through the same audited envelope already used for protected changes and promotion apply.

### Admin UX

- In `/admin/flags/:key/rollouts`, each stage row gets a `Guardrails` cell: `Healthy`, `Pending data`, `Breached`, or `Muted`.
- The sidebar shows a compact card per signal: metric name, threshold, last refresh, sample size, and the decision it can trigger.
- The confirm modal for `Advance` adds one line: "Monitoring will watch 2 guardrails for 30m after stage start."
- The timeline renders a human sentence: "Rollback triggered automatically after `checkout_error_rate` breached threshold for 11m at rollout stage 25%."

This is additive to the current IA rather than a new area of the app.

## Idiomatic Elixir / Phoenix Shape

- `rulestead` should keep runtime evaluation pure and local. Mature platforms and GrowthBook's docs reinforce that local evaluation plus separate control-plane publication is the right shape.
- Use an explicit behaviour seam for metrics and guardrail decisions, the same way Rulestead already prefers host-owned seams for auth, tenancy, and notifier concerns.
- Keep every guarded mutation inside `Ecto.Multi` so the rollout state change, guardrail event, and audit event commit atomically. `Ecto.Multi` is the idiomatic Ecto transaction envelope for grouped repo operations.
- Keep mounted admin state URL-driven with router-mounted LiveViews and `handle_params/3`, which is idiomatic for shareable filter/scope state in Phoenix LiveView.
- Use `Phoenix.LiveView.stream/3` and async assigns for large audit and simulation views rather than custom polling-heavy UI state.
- Use Oban for scheduled checks only if the host already enabled the existing scheduling seam. Do not make runtime evaluation depend on background jobs.

## Lessons From Comparable Systems

### What they did right

- **LaunchDarkly:** guarded rollout is tied to explicit monitoring windows, metric deltas, and automatic rollback semantics. The useful lesson is not the stats engine; it is that rollback decisions are attached to rollout stages, not hidden in a separate observability product.
- **Unleash:** lifecycle and expected lifetime are first-class, and flag state is visible in the main product. Rulestead should copy this discipline whether or not guarded rollout ships next.
- **GrowthBook:** local SDK evaluation, typed values, explicit rule types, and a modular stance between feature delivery and experiment analysis are all directly relevant. Their "safe rollout" and sticky bucketing posture shows how to add rollout intelligence without making the runtime remote.
- **Flipper:** warns clearly against using "percentage of time" for gradual rollout because it causes inconsistent user experience. Rulestead should preserve its deterministic targeting-key model and never introduce non-sticky rollout semantics as the default.

### What they got wrong or where the footguns are

- **LaunchDarkly-class complexity:** too much metric and statistics sophistication too early would pull Rulestead out of its mounted-library sweet spot.
- **Tooling that hides data quality problems:** GrowthBook and LaunchDarkly both only work well when event quality is credible. A guarded rollout built on weak impressions or weak sample sizes becomes fake safety.
- **Boolean-era shortcuts:** Flipper's warning on time-based rollout and FunWithFlags-style precedence complexity both reinforce the same point: rollout behavior must stay predictable, stable, and explainable.
- **Lifecycle debt neglect:** vendors that treat cleanup as an afterthought create long-term operator drag. Rulestead should not postpone lifecycle so long that guarded rollout lands on top of unmanaged flag sprawl.

## DX and Host-App Friendliness

- The host must wire metrics through one explicit behaviour, not by sprinkling callback modules across runtime, admin, and jobs.
- There should be a single default: manual stage advancement plus optional guardrails. Auto-advance with auto-rollback is a later layer.
- The failure mode must be least-surprising:
  - missing metric data means `hold` or `pending`, never "assume healthy"
  - stale metric data means no stage advance
  - insufficient sample means no automatic rollback
- Operators need copyable evidence and deterministic replay: "why did rollout stage 3 roll back?" should be answerable from audit data alone.
- The `rulestead_admin` package should only render and orchestrate; the rollout decision engine belongs in `rulestead`.

## Requirement Themes If Chosen

- Guardrail signal seam with a safe no-op default
- Stage-level monitoring windows and freshness rules
- Hold / rollback decision policy with explicit reasons
- Audit and timeline vocabulary for guardrail-triggered actions
- Admin rollout status cards that surface health without pretending to be full observability
- Verification around false-positive handling, stale signals, insufficient sample, and tenant/env scoping

## Non-Goals If Chosen

- No built-in metrics warehouse
- No custom stats engine beyond simple threshold and freshness evaluation
- No org-wide rollout dashboards
- No tenant-partitioned metric storage model
- No standalone `rulestead_admin` control plane
- No automatic feature experimentation analysis bundled into the rollout milestone

## Recommendation

Defer `v1.2.0 — Guarded Rollout Foundations` by one milestone.

It is a credible near-term candidate, but not the best next milestone for Rulestead right now. The local planning docs already rank lifecycle and ownership ahead of it, and the external ecosystem evidence supports that ordering: Unleash-style lifecycle hygiene is the more universal daily need, while LaunchDarkly-style guarded rollout is a higher-complexity maturity layer. Rulestead has enough rollout and governance infrastructure that guarded rollout can land cleanly after lifecycle/ownership, and it will likely land better once owner metadata, stale-state workflows, and cleanup posture are first-class.

If the roadmap still wants to reserve this as the next big differentiator, the right framing is: ship lifecycle/ownership first, then make guarded rollout the immediately following milestone with a deliberately bounded v1 that only supports host-supplied guardrails, stage hold, and audited rollback.

## Guardrails

- Keep guarded rollout as a rollout extension, not a new product line.
- Keep metric ingestion host-supplied and behaviour-driven; do not bind core runtime to Prometheus, OpenTelemetry, or vendor APIs.
- Keep automatic actions fail-closed: no data, stale data, or low sample should stop automation.
- Reuse the existing governed-action and audit envelope; do not create a second mutation path.
- Preserve deterministic targeting and stable stickiness; never use time-based rollout semantics for user-facing release progression.
- Keep tenant and environment scope explicit in every guardrail query and every audit row.
- Do not widen `rulestead_admin` into fleet observability, org administration, or standalone control-plane territory.
